## 1. Исходная проблема

В первом ответе Ирины по нескольким ЖК модель иногда делает ответ формально правильным, но слабым для клиента:

- варианты склеиваются в один абзац вместо нормального списка;
- описательная часть звучит сухо или одинаково;
- модель пишет общие фразы вроде «понятен бюджет входа»;
- если дать модели больше свободы, она начинает рекламировать и додумывать;
- особенно рискованны инвестиционные сценарии: модель легко уходит в «выгодно», «доходность», «ликвидность», «перспективность», хотя таких фактов в MCP нет.

По контракту проекта нельзя выдумывать факты. Все объяснения должны опираться только на `facts[]` / `near[]` из MCP/search.

## 2. Что проверяли

Проверяли несколько архитектурных гипотез для блока короткого объяснения «почему этот ЖК подходит под запрос клиента».

### Гипотеза A — дать модели полный MCP и хороший prompt, пусть сама соберёт ответ

Результат: отклонено как основной путь.

Причина: это уже близко к текущей архитектуре. Она уже дала плохой prod-пример: по запросу `кварира для инвестуий` модель получила несколько фактов, но сжала 3 ЖК в одну фразу без нормального списка и без качественных причин.

Вывод: одной сильной инструкции недостаточно. Нужен кодовый контроль структуры и валидатор.

---

### Гипотеза B — код полностью сам пишет причины

Результат: частично подходит, но не как финальный UX.

Плюсы:

- безопасно;
- не выдумывает;
- легко валидировать.

Минусы:

- быстро появляются одинаковые фразы;
- текст становится шаблонным;
- сложно красиво объяснять компромиссы внутри одной подборки.

Вывод: код должен владеть фактами, структурой и финальным рендером, но модель полезна для выбора сравнительного угла.

---

### Гипотеза C — model reason-only: сценарий + факт → короткая причина

Результат: рабочая, но ограниченная.

Тесты с `scenario + object + fact_type + allowed_meaning + human_angle` дали безопасные ответы.

Проблема: если прогонять каждый ЖК отдельно, модель не видит всю подборку и часто повторяет один и тот же смысл:

- «понятный бюджет»;
- «с отделкой проще»;
- «метро рядом».

Вывод: для первого списка модель должна видеть все 2–3 варианта вместе.

---

### Гипотеза D — full comparative reasons: модель пишет готовые причины для всех ЖК

Результат: отклонено для MVP.

Плюсы:

- модель лучше видит сравнение внутри одного ответа.

Минусы:

- причины становятся длинными;
- JSON иногда ломается;
- модель начинает писать финальный клиентский текст, где выше риск рекламы и лишних смыслов.

Вывод: модель не должна писать готовый клиентский reason целиком.

---

### Гипотеза E — comparative angle-only

Формат:

```json
{
  "items": [
    {"idx": 1, "angle": "самый низкий вход по цене"},
    {"idx": 2, "angle": "выше по стартовой цене"},
    {"idx": 3, "angle": "средний уровень стартовой цены"}
  ]
}
```

Результат: технически стабильно, но сухо.

Плюсы:

- JSON стабильный;
- факты не выдумываются;
- хорошо видно различие между вариантами.

Минусы:

- мало подсказки для живого рендера;
- кодовому renderer сложно понять, как красиво подать угол.

Вывод: нужно добавить `tone`.

---

### Гипотеза F — comparative angle + tone

Формат:

```json
{
  "items": [
    {
      "idx": 1,
      "angle": "самый низкий вход по цене",
      "tone": "нейтральный, про бюджет"
    }
  ]
}
```

Результат: принято как безопасная схема для MVP.

Почему это лучший баланс:

- модель видит все варианты вместе;
- модель выбирает сравнительный смысл, а не пишет финальный рекламный текст;
- код остаётся владельцем финальной фразы;
- валидатор может проверить короткие поля;
- меньше риска сломанного JSON;
- меньше риска выдуманных фактов.

Итоговая схема:

```text
MCP facts
→ code selects visible options and known fields
→ model returns comparative angle + tone per option
→ code renderer writes final client phrase
→ validator checks safety, repeats, JSON, forbidden claims
→ final numbered answer
```

---

### Гипотеза G — angle + client_value + style_hint

Результат: отклонено.

Причина: формат слишком развязал модель.

Плохие эффекты в тестах:

- «премиальная локация» без достаточного основания;
- «Садовое кольцо» — выдуманный/неподтверждённый контекст;
- «сэкономить больше половины бюджета»;
- «диверсификация бюджета»;
- слишком рекламный инвестиционный тон.

Вывод: `client_value` и `style_hint` как поля ответа модели в MVP не используем.

## 3. Принятая архитектура MVP

### 3.1. Что делает код

Код отвечает за:

1. Получить `search_response` от MCP/search.
2. Выбрать максимум 3 видимых варианта.
3. Нормализовать поля:
   - `name`
   - `location`
   - `price_min` / `price_range`
   - `finishing`
   - `ready` / `status` / `delivered`
   - `metro`
   - `area`
   - `why_close`
4. Определить сценарий клиента:
   - `investment`
   - `family`
   - `self_use`
   - `budget`
   - `fast_move`
   - `metro_access`
5. Подготовить компактный payload для модели.
6. Получить `angle + tone`.
7. Отрендерить финальные человеческие фразы.
8. Проверить валидатором.
9. Собрать финальный numbered list `1. / 2. / 3.`.

---

### 3.2. Что делает модель

Модель не пишет финальный ответ клиенту.

Модель только выбирает сравнительный угол и тон подачи.

Выход модели строго:

```json
{
  "items": [
    {
      "idx": 1,
      "angle": "3-8 слов, смысл только из INPUT facts",
      "tone": "3-8 слов, как подать этот смысл"
    }
  ]
}
```

Пример для метро:

```json
{
  "items": [
    {
      "idx": 1,
      "angle": "минута до метро, срок раньше",
      "tone": "акцент на ежедневной дороге"
    },
    {
      "idx": 2,
      "angle": "минута до метро, цена выше",
      "tone": "спокойно обозначить высокий бюджет"
    },
    {
      "idx": 3,
      "angle": "минута до метро, цена ниже",
      "tone": "подчеркнуть доступный вход у метро"
    }
  ]
}
```

---

### 3.3. Что делает renderer

Renderer превращает `angle + tone` в безопасную живую фразу.

Например:

```json
{
  "angle": "минута до метро, цена ниже",
  "tone": "подчеркнуть доступный вход у метро"
}
```

Финальная фраза:

```text
До метро около минуты пешком, при этом старт по цене здесь ниже, чем у соседних вариантов.
```

Важно: renderer должен понимать семантику angle. Примитивный keyword renderer может испортить хороший angle.

---

## 4. Валидатор

Валидатор обязателен.

Он проверяет:

1. JSON валидный.
2. Есть item для каждого видимого ЖК.
3. `idx` совпадают с вариантами.
4. Нет повторяющихся angle/tone.
5. Нет запрещённых claims:
   - доходность;
   - аренда;
   - ликвидность;
   - рост цены;
   - окупаемость;
   - выгодная инвестиция;
   - инвестиционно привлекательный;
   - лучший;
   - идеальный;
   - максимально;
   - неподтверждённые школы, парки, дворы, видовые квартиры, скидки, ипотека.
6. Слова внутри `object_name` не считаются нарушением. Например, `парк` в `Бусиновский парк` или `Белая Дача парк` не банится.
7. `премиальный` / `премиальное` — warning-level, если это не подтверждено MCP-полем `class` / `new_building_class`.

Если модель дала плохой output:

1. Один retry с указанием ошибки.
2. Если снова плохо — fallback на кодовый шаблон.

## 5. Результаты проверки моделей

### 5.1. `google/gemini-3.1-flash-lite-preview`

Статус: usable backup / MVP fallback.

Плюсы:

- быстрый;
- JSON стабильный;
- angle-задачу понимает лучше, чем обычный reason-only.

Минусы:

- иногда тянется к словам про премиальность/статус;
- может быть менее точен в сравнении;
- требует хорошего renderer.

Вывод: можно держать как резервную быструю модель.

---

### 5.2. `google/gemini-3.5-flash`

Статус: лучший live-кандидат.

Плюсы:

- быстрый: примерно 3.45 секунды wall time в benchmark;
- хорошо понимает сравнение внутри подборки;
- даёт нормальные `angle + tone`;
- подходит для Telegram live UX.

Минусы:

- иногда суховат и аналитичен;
- иногда использует `премиальный`, это нужно ловить validator warning;
- если дать слишком свободный prompt, может уйти в рекламу.

Вывод: основной кандидат для live MVP.

---

### 5.3. `openai/gpt-5.4-mini`

Статус: лучшее качество, но слишком медленно для live.

Плюсы:

- самый чистый и безопасный angle/tone;
- особенно хорошо формулирует компромиссы `цена ↔ срок`;
- меньше рекламного шума.

Минусы:

- benchmark: около 19.53 секунды wall time;
- для Telegram первого ответа это слишком долго.

Вывод: использовать для offline tests / benchmark / возможно для не-live задач, но не как live default.

---

### 5.4. `openai/gpt-5.5`

Статус: хороший, но не live default.

Плюсы:

- качественно понимает сравнение;
- особенно хорошо отработал metro case;
- быстрее, чем `gpt-5.4-mini`.

Минусы:

- всё равно 10–13 секунд wall time;
- дороже Gemini;
- не дал настолько большого прироста, чтобы оправдать live latency.

Вывод: подходит для benchmark/offline, но не основной live-кандидат.

---

### 5.5. `openai/gpt-4o`

Статус: не рекомендован для этой задачи.

Плюсы:

- точный;
- безопасный.

Минусы:

- сухой;
- аналитический;
- не даёт выигрыша именно для коротких comparative angles.

Вывод: не использовать как основной вариант для reason layer.

4. `google/gemini-3.1-flash-lite-preview` — быстрый backup / fallback.
5. `openai/gpt-4o` — точный, но слишком сухой для этой задачи.

## 8. Что собираем дальше

### 8.1. MVP scope

Собираем не полный новый ответчик, а отдельный controlled reason layer для первого списка вариантов.

Первый MVP должен работать только там, где:

- есть 2–3 видимых ЖК;
- есть MCP facts;
- сценарий клиента понятен;
- нужно добавить короткую причину/угол для каждого варианта.

---

### 8.2. Компоненты MVP

#### 1. `scenario_detector`

Использовать уже существующие params/intent:

- `purpose=investment` → `investment`;
- `purpose=family` → `family`;
- запрос про метро → `metro_access`;
- бюджетные ограничения → `budget`;
- готовность/переезд/сдано → `fast_move`;
- иначе → `self_use`.

#### 2. `option_fact_normalizer`

Нормализует поля MCP:

- price;
- min/max price;
- finishing;
- ready/status/delivered;
- metro;
- area;
- location;
- why_close.

#### 3. `comparative_payload_builder`

Собирает для модели только безопасные данные:

```json
{
  "scenario": "metro_access",
  "user_query": "поближе к метро",
  "options": [
    {
      "idx": 1,
      "name": "Левел Академическая",
      "facts": {
        "metro": "Академическая (1 мин. пешком)",
        "price_min": "23.02 млн",
        "ready": "2025 г., 4 кв."
      }
    }
  ],
  "allowed_fact_types": ["metro", "price_min", "ready", "finishing", "area", "location"],
  "forbidden_claims": ["доходность", "аренда", "ликвидность", "рост цены", "выгодный", "идеальный"]
}
```

#### 4. `angle_tone_model_call`

Модель: сначала `google/gemini-3.5-flash`, temp `0.25`.

Возвращает:

```json
{
  "items": [
    {"idx": 1, "angle": "...", "tone": "..."}
  ]
}
```

#### 5. `angle_tone_validator`

Проверяет output модели.

#### 6. `reason_renderer`

Кодом превращает angle/tone в финальные фразы.

#### 7. `first_list_renderer`

Собирает финальный ответ:

```text
Нашла несколько вариантов рядом с метро.

1. Левел Академическая — от 23 млн, сдача в 2025 году.
   До метро около минуты пешком, а срок сдачи ближе, чем у остальных.

2. CULT — от 44,6 млн, сдача в 2026 году.
   До метро тоже около минуты пешком, но бюджет здесь заметно выше.

3. ГК «Варшавские ворота» — от 9,5 млн, сдача в 2026 году.
   Метро рядом, при этом старт по цене самый низкий в этой подборке.

Какой ЖК хотите рассмотреть подробнее?
```

## 9. Acceptance criteria

MVP считается готовым локально, если:

1. На 20–30 реальных MCP/search responses:
   - JSON модели валидный;
   - каждый visible option получил angle/tone;
   - нет запрещённых claims;
   - нет повторяющихся смыслов внутри одного ответа;
   - финальный ответ использует numbered list;
   - не больше 3 ЖК;
   - один финальный вопрос.
2. `python3 -m py_compile` проходит.
3. `scripts/nmbot_test_agent.py` relevant suite проходит.
4. На prod/VPS после deploy:
   - `novostroy-bot.service` жив;
   - live/prod smoke показывает новый формат;
   - `scripts/or_cost.py` выведен после проверки.

Важно: пока проверено только локально/через gateway experiments, нельзя говорить «готово в проде».

---

## 10. Что НЕ делаем в MVP

- Не даём модели писать весь первый ответ целиком.
- Не используем `client_value` / `style_hint` как выход модели.
- Не обещаем инвестиционную доходность, аренду, рост цены, ликвидность.
- Не выдумываем школы, парки, дворы, видовые квартиры, скидки, ипотеку.
- Не меняем сразу весь chat prompt.
- Не делаем deploy без локального теста и prod smoke.

## 12. Финальный вывод

Гипотеза подтверждена не как «модель сама красиво пишет», а как контролируемая связка:

```text
факты MCP + сценарий клиента + сравнительный анализ модели + кодовый renderer + валидатор
```

Лучший live-путь сейчас:

```text
google/gemini-3.5-flash, temperature=0.25, output=angle+tone, финальный текст пишет code renderer
```

Лучший offline/quality benchmark:

```text
openai/gpt-5.4-mini
```

Главный риск не в модели, а в renderer/validator: если renderer примитивный, он испортит хороший angle; если validator слабый, модель начнёт рекламировать или выдумывать.
