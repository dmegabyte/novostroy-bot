# nmbot Best Practices — source conspect

Дата: 2026-07-08

Этот документ — не место для догадок. Здесь только то, что прямо видно в прочитанных источниках: внутренние доки проекта и README внешних репозиториев.

## Правило документа

Для каждого пункта ниже есть четыре части:

1. **Что говорит источник** — только факт из прочитанного материала.
2. **Почему это best practice** — почему этот факт полезен на практике.
3. **Риски** — что может пойти не так.
4. **Как безопасно взять в nmbot** — как перенести идею без поломки контракта.

Если источник не содержит утверждение, мы его не добавляем.

## Как читать документ

- **Быстро понять главное** → смотри `0.1 Краткая таблица` и `5. Рабочий чеклист`.

- **Ищешь внешние идеи** → смотри разделы `1` и `10`, но перед внедрением снова читай README конкретного репозитория.

## 0. Приоритет источников

Если источники расходятся, порядок такой:

1. `docs/IDEAL_IRINA_UX.md` — UX-контракт и поведение Ирины.

## 1. Что видно в внешних ботах

### 1.1 OpenHouse Bot

**Источник:** README репозитория `masood1996-geo/openhouse-bot`.

**Что говорит источник**

- setup wizard для старта;
- `config.yaml` как центр конфигурации;
- filter pipeline;
- SQLite de-dup;
- continuous monitoring;
- anti-bot / captcha fallback;
- Telegram notifications;
- AI discovery новых источников.

**Почему это best practice**

- wizard снижает порог входа;
- pipeline делает поведение повторяемым;
- dedupe убирает повторный шум;
- fallback сохраняет работу при блокировках;
- monitoring показывает не только “бот жив”, но и “бот реально делает работу”.

**Риски**

- wizard может раздуться и отпугнуть новичка;
- fallback легко превратить в сложный костыль;
- антибот-слой может незаметно стать главным сценарием;
- если pipeline не формализован, он остаётся только в голове разработчика.

**Как безопасно взять в nmbot**

- держать onboarding коротким и линейным;
- выделить fallback как отдельный путь и логировать его включение;
- зафиксировать этапы pipeline в коде, логах и тестах;
- не позволять fallback менять правила основного UX.

### 1.2 Flathunter

**Источник:** README репозитория `flathunters/flathunter`.

**Что говорит источник**

- periodic scraping loop;
- конфиг как основной способ управления;
- уведомления в Telegram/Mattermost/Slack;
- proxy/captcha support;
- Docker / GCloud deployment;
- state of already seen (`processed_ids.db`).

**Почему это best practice**

- периодический цикл лучше ручного refresh;
- явный config проще проверять и деплоить;
- already-seen state делает продукт менее раздражающим;
- контейнеризация и cloud-путь уменьшают разрыв между dev и prod.

**Риски**

- конфиг может стать слишком широким и непонятным;
- слишком много каналов уведомлений размывает операционную картину;
- proxy/captcha — зона нестабильности;
- плохая синхронизация state даст дубли или ложные пропуски.

**Как безопасно взять в nmbot**

- держать config minimal first, advanced later;
- выбрать один основной канал в UX, остальные оставить техническими;
- добавить явную валидацию конфига;
- отдельно тестировать dedupe и миграции состояния.

### 1.3 Trovare

**Источник:** README репозитория `Val1t3/Trovare`.

**Что говорит источник**

- AI analysis на уровне ссылки;
- automated daily scraping;
- interactive chat для сравнения listings;
- обновление критериев на лету.

**Почему это best practice**

- пользователь не любит заново объяснять одно и то же;
- сравнение в чате экономит время;
- система полезнее, если умеет продолжать разговор, а не обнулять его.

**Риски**

- можно перепутать уточнение с новым сценарием;
- чат может уйти в бесконечный цикл уточнений;
- если сравнение не опирается на факты, появляется галлюцинация.

**Как безопасно взять в nmbot**

- хранить sticky topic;
- отделить `repeat_search` от operator handoff;
- сравнивать только фактические поля из источника;
- покрыть короткие follow-up ответы UX-тестами.

## 2. Что фиксируют внутренние доки nmbot

### 2.0 MCP — единственный источник фактов

**Что говорит источник**

`docs/PRODUCT_TZ.md` прямо говорит, что MCP — единственный источник фактов для клиентского ответа, а внутренние механизмы клиенту не раскрываются.

**Почему это best practice**

- клиент получает только проверяемые данные;
- меньше риска выдумать факт;
- проще сохранять доверие и воспроизводимость ответа.

**Риски**

- можно случайно выдать внутреннюю технику вместо ответа;
- можно начать “докрашивать” ответ общими знаниями.

**Как безопасно взять в nmbot**

- любые цифры и свойства показывать только из MCP/search-ответа;
- внутренние поля не выводить клиенту;
- технический фолбэк держать как отказ от подтверждения, а не как замену факту.

### 2.1 `docs/IDEAL_IRINA_UX.md`

**Что говорит источник**

- это единый UX-контракт;
- короткие follow-up вроде `да`, `все`, `все проверь` должны продолжать тему;
- операторская воронка — отдельный режим, а не случайный текстовый хвост.

**Почему это best practice**

- сохраняет диалог естественным;
- уменьшает повтор объяснений;
- помогает не терять контекст на коротких ответах.

**Риски**

- sticky-логика может удержать старую тему слишком долго;
- если в клиентский текст попадут внутренние токены, UX ломается.

**Как безопасно взять в nmbot**

- применять sticky только к коротким follow-up;
- не переносить сырые внутренние значения в видимый ответ;
- покрыть тестами `да`, `все`, `все проверь`, опечатки.

### 2.2 `docs/SCENARIO_MCP_CONTRACT.md`

**Что говорит источник**

- `repeat_search` — это новые варианты, а не повтор старого short-list;
- `repeat_search` должен исключать уже показанные варианты;
- `operator` — это короткое подтверждение и передача выбранного контекста, если он есть.

**Почему это best practice**

- не даёт пользователю снова видеть то же самое;
- разделяет повторный поиск и операторский handoff;
- делает сценарии предсказуемыми.

**Риски**

- `repeat_search` может превратиться в повтор старого списка;
- operator handoff может потерять выбранный контекст;
- если контракт не закреплён, сценарии начинают смешиваться.

**Как безопасно взять в nmbot**

- хранить separate state для `repeat_search` и `operator`;
- тестировать exclude-list;
- проверять, что operator-ветка не подменяет repeat-search.

### 2.3 `docs/IRINA_UX_RELEASE_CHECKLIST.md`

**Что говорит источник**

- release gate включает `py_compile`, `scene_router_test`, `h028`, `h029`, `ux_e2e`, `deploy`, а для поведения LLM/MCP/search/chat — ещё `dialog`.

**Почему это best practice**

- маленькая правка может сломать UX на нескольких слоях сразу;
- набор проверок ловит и детерминированные, и семантические регрессии;
- локальный green без prod gate даёт ложное чувство готовности.

**Риски**

- слишком длинный gate замедляет цикл;
- частично зелёные тесты могут выглядеть как “готово”, хотя это не так.

**Как безопасно взять в nmbot**

- держать gate в понятном порядке;
- явно писать, что защищает каждый suite;
- для UX-изменений не завершать работу без prod/VPS проверки.

### 2.4 `docs/RESPONSE_IMPROVEMENT_PLAN.md`

**Что говорит источник**

- сначала дешёвые проверки, потом дорогие LLM/MCP-прогоны;
- выбранный объект должен всё равно получать живой ответ от LLM;
- `option:<idx>` и `action:details:<idx>` — разные действия;
- кнопки после выбранного объекта должны отвечать последнему вопросу;
- после получения номера нужен нормальный финал, а не сухое `Номер принят`.

**Почему это best practice**

- экономит платные прогоны;
- не убивает живую подачу шаблоном;
- помогает не смешивать выбор объекта и раскрытие деталей;
- делает конец воронки человеческим.

**Риски**

- можно рано тратить деньги на полный gate;
- можно сделать карточку выбранного ЖК сухой;
- можно поставить не те кнопки и сломать следующий шаг.

**Как безопасно взять в nmbot**

- сначала `py_compile` и unit suites без LLM/MCP;
- платный dialog запускать только после локального green;
- разделять выбор объекта, детали и операторский шаг;
- следить, чтобы кнопки отвечали последнему вопросу.

### 2.5 `docs/PROMPT_ARCHITECTURE.md`

**Что говорит источник**

- main prompt управляет формой и безопасностью;
- scenario/facet prompts добавляют сценарный смысл;
- code prepares material / normalizes / validates;
- `response.message`, `response.items[].reason`, `response.question` пишет LLM.

**Почему это best practice**

- снимает путаницу между тем, кто готовит контекст, и тем, кто пишет клиентский ответ;
- не даёт коду стать автором красивого текста;
- делает границы слоёв проверяемыми.

**Риски**

- если правило не закрепить, код начнёт подменять LLM-ответ;
- сценарные факты могут случайно уехать в main prompt;
- можно повторить одно и то же в main и scenario layers.

**Как безопасно взять в nmbot**

- хранить facts в scenario/facet слоях, а не в `chat_v1.txt`;
- кодом только готовить материал и валидировать контракт;
- держать итоговый prompt как конструктор, а не как хвостовой append.

## 3. Сводка практик, которые стоит брать в nmbot

### 3.1 Явный pipeline

**Источник:** `docs/SCENARIO_MCP_CONTRACT.md`, `docs/IRINA_UX_RELEASE_CHECKLIST.md`, а также паттерны OpenHouse Bot и Flathunter.

**Что это**

Цепочка вида `intent → search → normalize → filter → dedupe → present → follow-up/operator` должна быть видна в коде, логах и тестах.

**Почему это best practice**

- проще понять, где возникла ошибка;
- легче писать regression tests;
- легче менять один слой, не ломая остальные;
- пользователю виден предсказуемый сценарий.

**Риски**

- можно формально раздробить код и оставить скрытый монолит;
- можно перепутать реальный pipeline с красивой схемой на бумаге;
- лишние этапы создают overhead.

**Как безопасно взять в nmbot**

- вводить этап только если у него есть отдельная ответственность;
- в логах писать, какой этап принял решение;
- каждый новый этап закреплять тестом.

### 3.2 Дедупликация и already-seen state

**Источник:** OpenHouse Bot, Flathunter, `docs/SCENARIO_MCP_CONTRACT.md`.

**Что это**

Бот должен помнить, что уже показал клиенту, и не повторять тот же shortlist без причины.

**Почему это best practice**

- уменьшает раздражение;
- повышает ощущение, что бот помнит контекст;
- делает `repeat_search` полезным, а не декоративным.

**Риски**

- можно случайно выкинуть релевантный вариант;
- слишком агрессивный exclude-list обеднит выдачу;
- при плохой синхронизации state бот начнёт считать, что уже всё показал.

**Как безопасно взять в nmbot**

- хранить `last_options` и `already_shown` отдельно;
- сделать fallback, если свежих вариантов мало;
- логировать причину исключения;
- тестировать повторный поиск на маленьких и больших наборах.

### 3.3 Понятный config/onboarding

**Источник:** OpenHouse Bot, Flathunter.

**Что это**

Понятный стартовый путь: что заполнить, что проверить, как поднять локально и как отличить local от prod.

**Почему это best practice**

- снижает порог входа;
- уменьшает ошибки запуска;
- сокращает время до первого результата;
- делает поддержку дешевле.

**Риски**

- wizard может стать слишком длинным;
- в конфиг можно спрятать слишком много магии;
- человек может думать, что все параметры обязательны.

**Как безопасно взять в nmbot**

- разделить mandatory и advanced settings;
- валидировать обязательные поля;
- держать документацию рядом с конфигом;
- не превращать onboarding в анкету на десять минут.

### 3.4 Наблюдаемость и smoke checks

**Источник:** OpenHouse Bot, Flathunter, `docs/IRINA_UX_RELEASE_CHECKLIST.md`.

**Что это**

Health/status, свежие логи, smoke test, счётчики повторов и уходов в оператора.

**Почему это best practice**

- живой PID не гарантирует живой UX;
- без smoke тяжело понять, что именно сломалось;
- метрики делают качество измеримым;
- легче ловить регрессии после правок.

**Риски**

- можно утонуть в логах и метриках;
- если мониторинг слишком шумный, его начнут игнорировать;
- smoke без реального сценария не ловит UX-bug.

**Как безопасно взять в nmbot**

- хранить только полезные метрики;
- иметь один основной health-check и один более глубокий smoke;
- мерить не только размер payload, но и скорость ответа: размер запроса берём из
  `logs/model_payload_metrics-YYYY-MM-DD.jsonl`, а реальное время модели — из
  Overmind task list `/api/v1/tasks/api/list?limit=1000` как
  `completed_at - started_at` плюс очередь `started_at - created_at`;
- в health-check выводить только metadata: task id, timestamps, duration,
  queue, query/system-prompt длины; не печатать prompt text, client text и токены;
- smoke гонять по реальному пути;
- не расширять мониторинг без конкретного вопроса “зачем”.

### 3.4.1 Release gate как обязательная практика

**Источник:** `docs/IRINA_UX_RELEASE_CHECKLIST.md`.

**Что это**

Правка UX Ирины не считается готовой, пока не пройдены `py_compile`, `scene_router_test`, `h028`, `h029`, `ux_e2e`, `deploy`, а при изменениях LLM/MCP/search/chat — ещё `dialog`.

**Почему это best practice**

- защищает от частичных побед;
- ловит регрессии на нескольких слоях сразу;
- не даёт спутать локальный успех с prod-готовностью.

**Риски**

- gate может показаться длинным;
- часть проверок может соблазнить назвать работу готовой раньше времени.

**Как безопасно взять в nmbot**

- держать gate как обязательный путь для UX-изменений;
- не сокращать его “ради скорости”;
- отдельно отмечать, на каком шаге упало.

### 3.5 Sticky follow-up и семантическое продолжение

**Источник:** `docs/IDEAL_IRINA_UX.md`, `docs/SCENARIO_MCP_CONTRACT.md`, Trovare.

**Что это**

Короткие ответы вроде `да`, `все`, `проверь все`, `все проверь` должны продолжать тему, а не открывать новый сценарий.

**Почему это best practice**

- человек не любит заново объяснять контекст;
- диалог выглядит естественно;
- бот лучше работает с реальной речью, а не только с “чистыми” запросами.

**Риски**

- можно случайно удержать старую тему слишком долго;
- sticky logic может увести в неправильный сценарий;
- внутренние state tokens могут вылезти в ответ.

**Как безопасно взять в nmbot**

- делать sticky только для коротких follow-up;
- хранить тему, но не тащить сырые внутренние значения в клиентский текст;
- покрыть тестами `все проверь`, `да`, `ок`, опечатки;
- отделить routing decision от wording layer.

### 3.6 Разделение слоёв

**Источник:** `docs/SCENARIO_MCP_CONTRACT.md`, `docs/IRINA_UX_RELEASE_CHECKLIST.md`, `docs/IDEAL_IRINA_UX.md`.

**Что это**

Search, presentation, follow-up, operator handoff, monitoring и tests должны быть разделены.

**Почему это best practice**

- проще тестировать;
- проще менять отдельный кусок;
- легче объяснить поведение коллегам и будущему себе;
- меньше шансов случайно поломать соседний слой.

**Риски**

- чрезмерное дробление делает код трудно читаемым;
- слои могут начать копировать друг друга;
- интерфейсы между слоями могут стать слишком тяжёлыми.

**Как безопасно взять в nmbot**

- вводить слой только если у него есть своя ответственность;
- держать контракты короткими и явными;
- писать regression tests на границах слоёв;
- не добавлять абстракцию “на будущее”, если она не нужна сейчас.

### 3.7 Fallback / anti-bot как запасной путь

**Источник:** OpenHouse Bot, Flathunter.

**Что это**

Headless, proxy, captcha, alternate fetch — но только как резерв.

**Почему это best practice**

- реальные источники иногда блокируют;
- fallback повышает устойчивость;
- без него часть выдачи просто исчезнет.

**Риски**

- fallback может стать сложнее основного пути;
- можно начать чинить не UX, а только обход блокировок;
- пользователю легко показать техническую боль.

**Как безопасно взять в nmbot**

- сначала стабилизировать основной путь;
- fallback активировать только при необходимости;
- логировать причину переключения;
- не менять пользовательский сценарий из-за технического обхода.

### 3.8 Интерактивное уточнение критериев

**Источник:** Trovare, `docs/IDEAL_IRINA_UX.md`.

**Что это**

Пользователь должен легко попросить “ещё похожие”, “другие варианты”, “сравни”, “с отделкой”, “до 12 млн”.

**Почему это best practice**

- сокращает длину диалога;
- делает подбор похожим на работу живого консультанта;
- повышает полезность бота без роста интерфейсной сложности.

**Риски**

- можно превратить всё в бесконечный уточняющий цикл;
- если контекст хранится плохо, ответы начнут противоречить друг другу;
- без тестов появится хаос в follow-up логике.

**Как безопасно взять в nmbot**

- жёстко хранить последний список;
- обновлять критерии инкрементально;
- ограничивать число шагов без нового фактического сигнала;
- проверять follow-up тестами.

## 4. Карта сценариев из `SCENARIO_MCP_CONTRACT`

Эта карта нужна, чтобы не потерять важные сценарии при переносе best practices.

| Сценарий | Что должен делать | Риск, если забыть | Что брать в nmbot |
|---|---|---|---|
| `family` | Дать семейную пользу и evidence | Слишком общий продающий текст | Семейные поля должны быть приоритетом |
| `investment` | Показать вход, а не обещать доходность | Обещания доходности без фактов | Не выдумывать ROI и окупаемость |
| `rental` | Фокус на компактность, отделку, метро, готовность | Расплывчатый ответ без rental evidence | Проверять demand только по фактам |
| `search` | Показать точные и близкие варианты | Потеря обязательных условий | `near` должен быть объяснён |
| `repeat_search` | Показать другие варианты | Повтор старого списка | Exclude уже показанного |
| `explain_selection` | Объяснить выбор по уже показанным ЖК | Ответ без evidence | Работать по visible options |
| `fact_check` | Проверить один конкретный факт | Уход в продажу вместо проверки | Коротко да/нет/не подтверждено |
| `default` | Уточнить, если данных мало | Рано показывать ЖК | Один короткий вопрос вместо shortlist |
| `operator` | Передать человеку с контекстом | Придумать юридические/бронь условия | Коротко подтвердить и передать контекст |
| `off_topic` | Вернуть к недвижимости | Уход на внешнюю тему | Вежливый отказ без лишних деталей |

Почему это best practice: сценарии уже заданы контрактом, и без этой карты легко начать смешивать их между собой.

Риск: если таблицу читать как красивую справку, а не как контракт, она не защитит от регрессий.

Как безопасно взять в nmbot: использовать карту как чеклист при изменении routing/prompt/tests.

## 5. Рабочий чеклист для nmbot

Это короткая версия всего документа. Если меняем UX, routing, prompt, search, operator flow или state, проверяем эти пункты:

1. **Факт есть в источнике?**
   - Да → можно использовать.
   - Нет → не добавлять в клиентский ответ и не записывать как best practice.

2. **Слой выбран правильно?**
   - Search собирает факты.
   - Code normalizes / validates / prepares material.
   - LLM пишет клиентский текст.
   - Operator handoff не смешивается с `repeat_search`.

3. **Память диалога не шумит?**
   - `repeat_search` исключает уже показанное.
   - Sticky follow-up работает только для коротких продолжений.
   - В клиентский текст не попадают внутренние токены.

4. **Fallback не стал основным сценарием?**
   - Anti-bot / proxy / captcha / alternate fetch — только резерв.
   - Причина fallback логируется.
   - Пользовательский UX не меняется из-за технического обхода.

5. **Проверки идут от дешёвых к дорогим?**
   - Сначала `py_compile` и unit suites без LLM/MCP.
   - Потом точечные проверки изменённого слоя.
   - Потом `dialog`, если затронуты LLM/MCP/search/chat ответы.
   - Потом `deploy` и prod/VPS gate.

6. **Логи не раскрывают секреты?**
   - HTTP-клиенты (`httpx`, `httpcore` и аналоги) не должны писать INFO-запросы с URL, где может быть Telegram/API token.
   - Для runtime-бота шумные transport-логгеры держим не ниже `WARNING`.
   - Если секрет уже попал в лог, сначала редактируем/ротируем лог без вывода секрета в терминал, потом перезапускаем сервис и проверяем, что новые строки больше его не содержат.

7. **Практика считается внедрённой только если есть:**
   - кодовая правка;
   - тест/регрессия;
   - обновлённая документация;
   - понятный risk / mitigation note;
   - проверка на local + prod gate.

## 6. Анти-паттерны

- лечить routing/state баги только промптом;
- смешивать search, presentation и operator в одном месте;
- делать антибот главным сценарием;
- повторять те же ЖК в `repeat_search` без причины;
- позволять внутренним токенам попадать в клиентский текст;
- позволять transport-логам писать секреты в runtime-логи;
- считать живой PID доказательством готовности UX;
- добавлять внешнюю идею без повторного чтения её README и сверки с внутренними контрактами.

## 7. Куда писать знания

- **В репозиторий:** решения по UX/routing, новые сценарии, проверки, регрессии, backlog, `why/risk/mitigation` для конкретных изменений.
- **Наружу ссылкой:** внешние репозитории, сторонние docs по scraping / anti-bot / deployment, библиотечные API-спеки.

## 8. Короткая формула

Хорошая практика для nmbot — это то, что опирается на источник, уменьшает риск регрессий, понятно в поддержке и делает диалог короче, честнее и полезнее.

## 9. Out of scope

Этот документ не про:

- написание нового боевого кода;
- выбор библиотек без отдельного запроса;
- архитектуру вне prompt / UX / сценариев подбора;
- спор о вкусе формулировок без ссылки на источник;
- общие советы “как везде принято”, если их нет в прочитанных доках или README.

Если нужен новый слой или спорная рекомендация — сначала ищем источник, потом добавляем сюда.
