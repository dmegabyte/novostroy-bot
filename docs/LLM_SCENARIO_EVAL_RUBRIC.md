# LLM Scenario Eval Rubric

Источник истины для оценки LLM-first семейного / инвестиционного / rental / поискового / repeat_search / explain_selection / fact_check / refine_search / default / операторского / off-topic поведения.

Текущий зафиксированный результат: family-сценарий уже работает как сравнительная мини-презентация из 3 ЖК, где оркестратор выбирает сценарий, сценарий несёт MCP-профиль и subprompt, MCP возвращает карточку фактов, а LLM пишет живой ответ.

Этот документ отвечает на 3 вопроса:

1. Что тестируем.
2. Как тестируем.
3. Какой результат считаем правильным.

Если правило есть здесь, именно его используем в симуляторе и ручной проверке.

---

## 1. Архитектурный контракт

Работаем по цепочке:

`scenario/orchestrator -> facets -> MCP query profile -> compact facts card -> chat LLM -> local check`

Коротко это выглядит так:

`запрос пользователя -> определение основного сценария -> добавочные facets -> выбор MCP-профиля -> MCP-карточка -> subprompt -> ответ пользователю`

### Главная цель

Научить систему **правильно формировать MCP-запросы под сценарий** и **собирать хороший пользовательский ответ из реальных фактов**, без выдумок и без технической болтовни.

### Обязанности

- **Модель / orchestrator** определяет семантику запроса: family, investment, rental, search, repeat_search, explain_selection, fact_check, refine_search, default, operator, off-topic.
- **Facet layer** добавляет поперечные рубрики, которые не заменяют основной сценарий: например `mortgage` поверх family/search/investment.
- **Код** выбирает готовый MCP query profile по сценарию + facets и собирает request payload (`need`, `exclude`, `purpose`, `facets`, `mortgage_type`, и т. п.).
- **MCP/search** возвращает компактную карточку фактов.
- **Chat LLM** пишет человеко-читаемый ответ только по этой карточке.
- **Local checker** читает request payload, карточку и ответ, оценивая все слои по критериям ниже.

### Контракт артефактов live-run / Google Sheets

В live-run строках и таблицах строго разделяем две разные сущности:

- **`mcp_request`** — контракт запроса: что система попросила у MCP/search.
- **`mcp_response`** — фактический ответ MCP/search: только данные из блока `Поисковые факты`, то есть `facts[]`, `near[]`, `missing`, `params`.

`mcp_request` сам по себе **не является MCP-ответом**. Если CLI-прогон упал после печати `MCP-запрос:` и до блока `Поисковые факты:`, то в подготовленной строке `mcp_response` должен быть пустым объектом `{}`, а не эхо-копией запроса. Такая строка должна иметь явные предупреждения, например `cli_exit_nonzero:<code>` и `mcp_response_missing_after_request`.

Правильная интерпретация:

- `mcp_request` есть, `mcp_response={}` → MCP/search payload не был получен или не был распарсен; это технический сбой прогона/пайплайна, а не пустой осмысленный ответ MCP.
- Если `mcp_response_missing_after_request` есть в warnings, такой прогон считается **score = 0**: MCP/search не довёл payload до блока `Поисковые факты`, и это нужно чинить на уровне сценария / transport / parser.
- `mcp_response` содержит `facts/near/missing/params` → есть реальный MCP/search payload, и только его можно считать источником фактов для ответа клиенту.
- Google Sheets publisher не должен додумывать или чинить факты; он только отображает уже подготовленные поля валидатора.

### Запрещено

- определять смысл regex-ом там, где нужен сценарий;
- выдумывать MCP-данные: работаем только с фактами, которые реально пришли из MCP/search/card;
- если факта нет в MCP/search/card, его нет и в ответе;
- выдумывать факты не из карточки;
- проверять качество через внешние запросы валидации;
- уходить в технические термины в клиентском ответе (`regex`, `mcp`, `json`).

### Главный принцип изменений

- не делаем заплатку на каждую новую проблему отдельно;
- сначала пытаемся решить вопрос через сценарий, prompt и MCP-карточку;
- код используем только для выбора профиля, safety boundary и передачи данных;
- чем ближе изменение к тексту ответа, тем сильнее предпочитаем LLM-слой;
- код пишем только там, где нужна жёсткая защита, транспорт или нормализация фактов.

### Где что хранится

- **Prompt** — стиль, тон, формат ответа, запреты, правила по family/invest/search/repeat_search/explain_selection/fact_check/refine_search/default/operator/off-topic. Shared prompt хранит только универсальные правила; сценарные акценты живут отдельно в `prompts/scenarios/<scenario>_v1.txt`.
- **Universal card presentation rule** — общий стандарт того, как показывать карточку живо и сравнительно; все сценарные overlays должны ему соответствовать: `docs/CARD_PRESENTATION_RULE.md`.
- **Scenario** — в оркестраторе хранится только имя основного сценария; сам сценарий-пакет включает MCP query profile и subprompt.
- **Facet** — добавочный слой поверх сценария. Facet хранит свои MCP-поля и ограничения, но не забирает на себя весь ответ. Пример: `prompts/facets/mortgage_v1.txt`.
- **MCP query profile** — конкретный запрос к базе/схеме: какие таблицы и поля читать, какие фильтры ставить, какие карточки возвращать.
- **Card / facts** — уже готовые проверенные факты для ответа.
- **Code** — только оркестрация, вызов профиля, safety, fallback, local check.

### Как мы меняем систему

1. Сначала проверяем, можно ли решить через prompt.
2. Если prompt не хватает — меняем scenario/profile.
3. Если не хватает данных — расширяем MCP/card.
4. Код меняем только если без этого нельзя безопасно или стабильно доставить результат.
5. Если локальная проверка или promptfoo дают `watch` / `bad`, включаем **fix loop**: promptmaster может переписать prompt, scenario subprompt или MCP query profile, после чего проверку запускаем ещё раз.
6. Исправление — это нормальная часть цикла, а не отдельный ручной обход.
7. `good`-результаты не гоняем повторно: если слой уже зелёный, он остаётся baseline и в цикле не трогаем его без новой причины.
8. Для каждого сценария сначала формулируем, что важно покупателю и по каким параметрам искать, а уже потом строим MCP query profile и subprompt.

### Порядок сборки нового сценария

Перед тем как собирать любой новый сценарий, сначала явно отвечаем на вопрос: **какой MCP-запрос нужен в этом случае**.

Минимальный порядок такой:

1. Понять семантику запроса.
2. Решить, какие факты реально нужны в MCP/card.
3. Выбрать MCP query profile и только потом писать subprompt.
4. В subprompt обязательно добавить один хороший пример и один плохой пример презентации.
5. Проверить, что сценарий не требует фактов, которых нет в схеме.
6. Если MCP-профиль слабый, сначала исправить его, а не красивить ответ.

### Шаблон сборки сценария

Когда собираем новый сценарий, сначала фиксируем его как короткую карточку из 4 частей:

1. **Сильные стороны** — что именно этот сценарий должен показать клиенту.
2. **Что не говорим** — какие фразы, цифры и обещания запрещены.
3. **MCP-запрос** — какие поля и таблицы должны подтвердить сильные стороны.
4. **Subprompt** — как LLM должна красиво и честно подать эти сильные стороны, не выходя за рамки запретов.

Правило: сначала смысл и ограничения, потом MCP-профиль, потом subprompt.

### Как собирать параметры запроса

Параметры запроса не берём из головы. Сбор всегда идёт в таком порядке:

1. **Сигналы из текста клиента.** Что уже прямо сказано: комнаты, бюджет, район, метро, отделка, срок сдачи, «для семьи», «под инвестиции», «под сдачу», «другие варианты», «эти не подходят».
2. **Обязательные параметры сценария.** Каждый сценарий добавляет свои опоры: family → schools/kindergartens/parks/yard_without_cars; investment → entry/mortgage/egrn/counter_novos/compact_lots; rental → compact/finishing/metro/ready/demand; repeat_search → exclude уже показанные ЖК; default → не ищем, а уточняем.
3. **Добавочные facets.** Если клиент добавил поперечное условие, оно подключается поверх primary scenario. Пример: `family + mortgage` = family-опоры + ипотечные поля, а не выбор между family и mortgage.
4. **MCP-контракт.** В query кладём только те поля, которые реально поддерживает `docs/NOVOSTROYM_MCP_SCHEMA.md`. Если поля нет в контракте, его нельзя требовать и нельзя придумывать.
5. **Нормализация в профиль.** Из собранных сигналов формируем query profile: `purpose`, `facets`, `mortgage_type`, `need`, `exclude`, `rooms`, `max_price`, `district`, `floor`, `has_renovation`, `count`, `selected_option_name` и т.п. — но только если это оправдано сценарием и MCP.
6. **Проверка качества.** Если обязательных сценарных или facet-опор в query/profile нет, это уже проблема MCP/search layer, а не только response.

Базовое правило:
- клиент сказал мало → добавляем сценарные обязательные опоры;
- клиент сказал много → не теряем важное при нормализации;
- MCP не умеет это поле → не выдумываем его, а помечаем как gap;
- default сценарий → сначала один короткий вопрос, без поиска.

### Правило против перегруза промптов

- **Shared prompt держим коротким.** В общем файле оставляем только универсальные правила: тон, формат, запреты, один вопрос, без техслов.
- **Сценарный overlay — только про свой сценарий.** В `prompts/scenarios/<scenario>_v1.txt` кладём только то, что действительно отличает family / investment / rental / search / repeat_search / explain_selection / fact_check / refine_search / default / operator / off-topic.
- **Сценарный overlay сначала описывает buyer goal и search params.** В начале каждого `prompts/scenarios/<scenario>_v1.txt` фиксируем: что важно покупателю, по каким параметрам искать, потом MCP-запрос, потом subprompt.
- **Не дублируем одно и то же в двух местах.** Если правило уже есть в `docs/CARD_PRESENTATION_RULE.md` или в rubric, не копируем его ещё раз в shared prompt и в overlay.
- **Одна мысль — один уровень.** Если правило относится ко всем сценариям, оно живёт в rubric/card standard; если относится к одному сценарию — живёт только в его overlay.
- **Overlay не превращаем в полотно.** Сценарный файл должен быть коротким: цель, что показываем, чего не говорим, MCP-поля, шаблон мини-презентации, хороший и плохой пример.
- **Если правило повторяется в нескольких сценариях, выносим его в общий стандарт.** Лучше один универсальный источник, чем копия в каждом файле.
- **Если в ответе несколько ЖК, каждый ЖК должен быть отдельным блоком.** Между блоками — пустая строка, чтобы ответ легко читался, парсился и считался.
- **Сухой сплошной текст — это уже не 100.** Даже при правильных фактах и хорошем score ответ обязан оставаться живым, с явной границей между объектами.
- **Слишком короткий reason — тоже проблема.** Если `reason` у объекта выглядит как ярлык или подпись, а не как объяснение пользы, это не good-answer уровень.
- **`reason` = presentation_reason.** В JSON поле остаётся `reason`, чтобы не ломать парсер и таблицу, но по смыслу это мини-презентация: подтверждённый факт → отличие от других → польза клиенту сейчас. Список фактов через запятую (`сдан, с отделкой, парк рядом`) не считается хорошим `reason`.

## 2. Что тестируем

### 2.1 Family

Проверяем, что ответ:

- понимает, что запрос семейный;
- показывает реальную пользу для семьи;
- если в карточке есть 3 подходящих ЖК, выдаёт красивую презентацию именно по 3 ЖК;
- использует только подтверждённые MCP-факты;
- не делает сухой список.

Нужные факты, если они есть в карточке:

- школы;
- детские сады;
- парки / зелёные зоны;
- двор без машин;
- детские и спортивные площадки;
- безопасность / территория;
- метро / транспорт / экология.

### 2.2 Investment

Проверяем, что ответ:

- говорит о цене и пороге входа;
- использует ипотеку / ЕГРН / скидки только если они есть в карточке;
- если в карточке есть несколько ЖК, показывает 2–3 варианта с разными инвестиционными акцентами;
- если в карточке есть `ads` / `apartment_types`, показывает 1–2 конкретных компактных лота (студия, однушка, евро-формат) как реальный вход;
- если запрос звучит как “под сдачу в аренду”, оценивает объект как арендо-пригодный вход: компактный формат, отделка, метро, быстрая готовность, район, подтверждённый спрос;
- не обещает доходность;
- звучит как живой совет, а не техотчёт.

### 2.2.1 Mortgage facet / ипотека

Проверяем не как отдельный сценарий, а как добавочный слой поверх primary scenario.

Если клиент говорит «семейная ипотека», «IT-ипотека», «льготная ипотека», «первоначальный взнос», «ставка», «рассрочка»:

- primary scenario сохраняется: family остаётся family, search остаётся search, investment остаётся investment;
- request содержит `facets:["mortgage"]`;
- если тип понятен, request содержит `mortgage_type`: `family_mortgage`, `it_mortgage` или `subsidized_mortgage`;
- `need` содержит ипотечные опоры: `mortgage_calc`, `mortgage`, `discount`, `payment_by_installments`, `price`;
- ответ говорит только проверенные ипотечные условия из MCP/card;
- если MCP не вернул ставку/взнос/программу, ответ не придумывает цифры и не обещает одобрение.

Пример смешанного запроса:

`У меня семья, хочу семейную ипотеку` → primary scenario `family` + facet `mortgage`.

### 2.3 Rental / leaseability

Проверяем, что ответ:

- если клиент говорит «под сдачу в аренду», показывает арендо-пригодный вход без ставки и без обещаний доходности;
- делает акцент на компактности, отделке, метро, быстрой готовности, районе и подтверждённом спросе;
- если в карточке есть несколько ЖК, показывает 1–3 варианта с разными арендными акцентами;
- если в карточке есть `ads` / `apartment_types`, показывает 1–2 компактных лота как понятный вход;
- не называет точную ставку аренды и не обещает окупаемость;
- звучит как живой, продающий совет, а не техотчёт;
- не перегружает ответ цифрами, метриками и сухими перечислениями.

### 2.4 Search / shortlist

Проверяем, что ответ:

- показывает 1–3 варианта;
- остаётся коротким и понятным;
- держит один финальный вопрос;
- не перегружает списком.

### 2.5 Repeat search / новые варианты

Проверяем, что ответ:

- понимает, что клиент просит другие или новые варианты;
- не повторяет тот же shortlist без причины;
- показывает свежие варианты с новым сравнительным акцентом;
- сохраняет короткий и понятный формат;
- держит один финальный вопрос.

### 2.6 Default / нераспознанный сценарий

Проверяем, что ответ:

- не выдумывает сценарий, если смысл не распознан;
- задаёт один короткий уточняющий вопрос;
- не начинает поиск, пока сценарий неясен;
- остаётся вежливым и коротким.

### 2.7 Explain selection / почему подобрали именно так

Проверяем, что ответ:

- объясняет логику подбора по фактам текущей карточки;
- не делает новый поиск, если фактов в памяти достаточно;
- просит MCP enrich, если reason пустой, короткий или без доказательств;
- не раскрывает техническую кухню и не говорит «алгоритм решил»;
- отвечает живым языком: критерий → факт → польза для клиента.

### 2.8 Fact check / проверка утверждения

Проверяем, что ответ:

- явно говорит: подтверждено / не подтверждено / нет подтверждения в карточке;
- делает MCP check/enrich, если проверяемого поля нет в памяти;
- не додумывает скидки, окна на две стороны, ипотеку, метро, парковку, этажи, планировки;
- если поле не поддерживается MCP-контрактом, помечает gap и предлагает проверку по конкретному ЖК.

### 2.9 Refine search / уточнение условий

Проверяем, что ответ:

- понимает, что клиент меняет фильтр, а не просто спрашивает объяснение;
- сохраняет полезный прошлый контекст;
- собирает merged MCP profile из старых params и новых условий;
- разделяет проверяемые MCP-поля и непроверяемые желания;
- если условия слишком жёсткие, предлагает ослабить один параметр, а не выдумывает совпадения.

### 2.10 Operator

Проверяем, что ответ:

- мягко переводит к оператору, если нужны live-детали;
- не пытается выдумать наличие / бронь / этаж / корпус;
- сохраняет текущий контекст.

### 2.11 Off-topic

Проверяем, что ответ:

- вежливо ставит границу;
- возвращает к подбору новостроек;
- не уходит в объяснения системы.

## 3. Как тестируем

### Источник теста

Используем только локальный model output или simulator output. Без новых outbound-запросов для проверки качества.

### Шаги проверки

1. Берём сценарий.
2. Берём его MCP query profile / facts card.
3. Смотрим ответ модели.
4. Проверяем по чеклисту ниже.
5. Записываем cost.
6. Если тест не прошёл, возвращаемся в fix loop: promptmaster правит prompt/subprompt/profile и прогоняем проверку повторно.

### Формат проверки

Проверка всегда локальная, с чтением ответа по 4 слоям:

- facts;
- completeness;
- beauty;
- scenario fit.

## 4. Чеклист оценки

### 4.1 Facts

ОК, если:

- все конкретные факты взяты только из карточки;
- нет выдуманных цен, сроков, школ, парков, метро, доходности;
- нет фактов, которых не было в MCP/card.

FAIL, если:

- есть хоть один факт из головы;
- есть техническая болтовня вместо фактов;
- есть утверждение, которого нет в карточке.

### 4.2 Completeness

ОК, если:

- family search/card содержит хотя бы часть реальных family-фактов (школы, сады, парки, двор без машин), если они доступны в MCP;
- family показывает реальную пользу для семьи;
- investment показывает цену / ипотеку / ЕГРН, если это есть;
- search показывает короткий shortlist;
- если search уже дал 3+ подходящих варианта (facts+near), ответ должен сохранить 3 явных карточки, а не схлопнуть shortlist до одной строки;
- operator даёт мягкий handoff;
- off-topic даёт короткую границу;
- есть один полезный следующий шаг.

FAIL, если:

- family сухой и без семейной пользы;
- family search/card не принёс family facts, хотя сценарий family;
- инвестиция без финансовых фактов, когда они были в карточке;
- shortlist слишком длинный;
- нет next step.

### 4.3 Beauty

ОК, если:

- ответ звучит по-человечески;
- 2–4 коротких абзаца или компактный список, если это shortlist;
- ровно один финальный вопрос;
- нет `regex`, `mcp`, `json` и прочей технички;
- tone friendly, calm, live-consultant.

FAIL, если:

- ответ сухой или плотный;
- family-ответ выглядит как технический список, а не как совет живого консультанта;
- family-ответ использует шаблон `Для семьи: ...` вместо объяснения пользы;
- family-ответ использует шаблон `Подходит семье: ...` вместо красивой презентации ЖК;
- каждый ЖК описан одной сухой строкой без мини-презентации;
- больше одного вопроса;
- слишком много пунктов;
- технический шум.

### 4.4 Scenario fit

ОК, если:

- family = семейная польза;
- investment = цена + финансы/ЕГРН;
- investment = 2–3 варианта, если карточка их даёт, и у каждого свой инвестиционный акцент;
- search = краткий shortlist;
- repeat_search = новые варианты вместо повторов;
- explain_selection = объяснение логики текущего подбора по фактам;
- fact_check = проверка конкретного утверждения через карточку/MCP;
- refine_search = изменение условий и новый merged MCP profile;
- default = один уточняющий вопрос без поиска;
- operator = мягкая передача;
- off-topic = вежливая граница.

FAIL, если сценарий перепутан.

### 4.5 Cost

ОК, если:

- cost всегда показан;
- ежедневный бюджет не превышен;
- целевой дневной лимит: **$4.00**.

### Как считаем cost

- всегда читаем реальный OpenRouter cost через `scripts/or_cost.py`;
- симулятор не использует заглушку `0.00` как итоговую стоимость;
- если `scripts/or_cost.py` недоступен, это ошибка окружения, а не нулевые расходы.

FAIL, если:

- cost не указан;
- дневной лимит превышен;

---

## 5. Обязательный формат оценки

Каждый раз проверяем ответ **по слоям**, а не одной общей оценкой.

### Формат отчёта

| Сценарий | Search / MCP-поиск | MCP card | Response | Verdict | Где проблема / что менять |
|---|---|---|---|---|---|

### Правило

- если search плохой — меняем запрос / профиль MCP;
- если search хороший, а card пустая — дорабатываем MCP query profile, нормализацию или форму карточки; факты не придумываем;
- если card хорошая, а response сухой — меняем presentation / salesiness / subprompt;
- если reason у объекта слишком короткий или похож на ярлык — тоже правим presentation / salesiness / subprompt;
- если response хороший, а сценарий перепутан — меняем выбор сценария;
- cost проверяем отдельно и всегда показываем рядом с verdict.

### Принцип

Одна итоговая оценка без слоёв не считается полной. Для каждого прогона должны быть видны хотя бы:

1. выбранный сценарий;
2. качество search/MCP;
3. качество карточки;
4. качество ответа;
5. итоговый verdict;
6. что именно править.
- проверка требует лишних внешних запросов.

---

## 6. Итоговые правила прохода

Результат считается успешным, если одновременно выполнены условия:

- Facts = OK
- Completeness = OK
- Beauty = OK
- Scenario fit = OK
- Cost = OK

Если есть хотя бы один FAIL — сценарий считается не прошедшим.

---

## 7. Ожидаемый результат по сценариям

### Family

Ожидаем:

- красивый топ-3 shortlist, если в карточке есть 3 подходящих ЖК;
- перед shortlist есть короткая логика выбора для семьи;
- каждый ЖК раскрыт как мини-презентация на 2–3 связных предложения;
- у каждого ЖК есть семейная польза из MCP, объяснённая человеческим языком;
- никакой выдумки про школы/сады/парки;
- живой совет, а не технический отчёт.

### Investment

Ожидаем:

- цена / входной порог;
- ипотека / ЕГРН / скидки, если они реально есть;
- при наличии `ads` / `apartment_types` — 1–2 конкретных компактных лота как понятный вход;
- без обещаний доходности.

### Search

Ожидаем:

- короткий shortlist с 2–3 мини-презентациями, если карточка даёт несколько ЖК;
- у каждого ЖК свой практический акцент (цена, отделка, готовность, метро, район);
- понятные варианты;
- один вопрос на выбор.

FAIL, если search превращается в голый список без презентации объектов.

### Repeat search

Ожидаем:

- клиент просит другие или новые варианты, а не повтор;
- shortlist обновлён и не копирует прежние ЖК без причины;
- у каждого ЖК свой новый акцент;
- короткое пояснение, чем новые варианты отличаются от прежних.

### Default

Ожидаем:

- один короткий уточняющий вопрос;
- без поиска до распознавания сценария;
- без выдумывания намерения клиента;
- без длинного объяснения системы.

### Operator

Ожидаем:

- мягкий handoff;
- сохранённый контекст;
- без выдуманных live-деталей.

### Off-topic

Ожидаем:

- короткую вежливую границу;
- возвращение к теме новостроек.

---

## 8. Пошаговая механика процесса

### Шаг 1. Оркестратор понимает сценарий

Модель / orchestrator решает, что это за запрос: family, investment, rental, search, repeat_search, default, operator или off-topic.

### Шаг 2. Сценарий выбирает MCP-профиль

Сценарий хранит:

- какой MCP/query профиль вызвать;
- какие поля нужны;
- какой subprompt дать ответной модели.

### Шаг 3. MCP возвращает facts card

MCP/search отдаёт только подтверждённые факты:

- ЖК / название;
- цена;
- район;
- отделка / готовность;
- family facts;
- finance facts;
- metro / transport / ecology.

### Шаг 4. LLM пишет ответ по карточке

В ответной LLM нет права:

- додумывать факты;
- писать технические термины клиенту;
- превращать family в сухой список.

### Шаг 5. Local checker оценивает ответ

Проверяем:

- facts;
- completeness;
- beauty;
- scenario fit;
- cost.

### Шаг 6. Если ответ не проходит — правим слой выше

Порядок правок такой:

1. prompt;
2. scenario / subprompt;
3. MCP/query profile или card shape;
4. code — только если нужна safety/transport/fallback.

---

## 9. Что считать хорошим выводом симулятора

Хороший вывод симулятора должен содержать:

- название сценария;
- query profile;
- compact facts card;
- ответ LLM;
- `QUALITY`;
- `COST`;
- итоговый `FINAL` summary.

## 10. Единая формулировка для работы команды

Если коротко:

1. Модель определяет сценарий.
2. Код выбирает профиль MCP-запроса.
3. MCP даёт карточку фактов.
4. LLM пишет ответ.
5. Локальный чеклист решает, прошёл ли результат.

### Практическое правило для команды

- если проблема про смысл — меняем сценарий или prompt;
- если проблема про недостаток фактов — меняем MCP query profile / card;
- если проблема про опасный формат или утечку технички — меняем code guard;
- если проблема повторяется в разных сценариях — ищем общий контракт, а не отдельную заплатку.

## 11. Как собирать новый сценарий

Если мы хотим добавить новый сценарий, идём по этому же шаблону:

### Шаг 1. Найти повторяющийся класс запросов

- смотреть реальные диалоги, симулятор, ручные заметки;
- понять, что это за семантика: новый тип запроса, а не случайный вариант старого.

### Шаг 2. Описать ScenarioProfile

Для сценария фиксируем:

- `name`;
- `purpose`;
- `query_tool`;
- `required_fields`;
- `answer_goal`;
- `scenario_prompt_path`.

### Шаг 3. Сделать отдельный scenario subprompt

- `prompts/scenarios/<scenario>_v1.txt`;
- только сценарные правила;
- без общей логики и без соседних сценариев.

### Шаг 4. Сформировать MCP query profile и card shape

- сначала читаем MCP-документацию по схеме/таблицам, которые относятся к сценарию;
- если неясно, какие поля реально возвращает MCP, делаем минимальный exploratory MCP-запрос и смотрим фактический payload;
- какие таблицы и поля нужны;
- какие facts обязательно возвращать;
- какой компактный card должен попасть в LLM.

### Шаг 5. Описать expected answer

- как должен звучать ответ;
- какой next step нужен;
- что считать dry list, FAIL, hallucination или слишком техничный тон.

### Шаг 6. Добавить checker rules

- facts;
- completeness;
- beauty;
- scenario fit;
- cost;
- scenario-specific fail-условия.

### Шаг 7. Прогнать simulator

- `python3 scripts/nmbot_scenario_sim.py --scenario <name>`;
- потом `--all`, если сценариев уже несколько;
- смотреть не только score, но и текст глазами.

### Шаг 8. Зафиксировать результат в доках

- `docs/LLM_SCENARIO_EVAL_RUBRIC.md`;
- `docs/CHANGELOG.md`;
- `progress.md`;
- при необходимости NotebookLM note.

### Что у нас уже есть для этого

- каноническая рубрика оценки;
- `PROFILE_REGISTRY` в `scripts/nmbot_scenario_sim.py`;
- `CARD_FACTORIES` в `scripts/nmbot_scenario_sim.py`;
- MCP schema doc (`docs/NOVOSTROYM_MCP_SCHEMA.md`) как источник правды по полям и таблицам;
- сценарные subprompt'ы для family / repeat_search / default и остальных overlays;
- проверка качества family на сухость, уникальные акценты и человеческую пользу;
- проверка короткого `reason` на уровне validator / promptmaster;
- учёт реального OpenRouter cost через `scripts/or_cost.py`;
- дневной лимит `$4.00`.

## 13. Как заполняем Google Sheet

Лист анализа — это уже не просто лог, а рабочая таблица для сравнения прогонов.

### Колонки листа `Ответ`

| Колонка | Что кладём |
|---|---|
| A | timestamp прогона |
| B | `версия` (`v2`, `v3`, ...) |
| C | `запрос` (сам текст user query) |
| D | `сценарий` (`family`, `investment`, `rental`, `search`, `repeat_search`, `default`, `operator`, `off_topic`) |
| E | `mcp` — только два верхнеуровневых блока: `mcp_contract` (request payload, который ушёл в поиск) и `mcp_response` (сырой search-MCP payload: `facts`, `near`, `missing`, `params` и прочие реальные поля). Никаких counts/names/verdict внутри этой колонки. |
| F | `ответ` — компактный JSON с response / visible_options / params / warnings / prompt_master_verdict |
| G | `оценка` — JSON verdict promptmaster (`score`, `verdict`, `problem_level`, `what_went_wrong`, `next_fix`) |
| H | `score` — просто число из `G.score` |

### Как это пишем

1. Сначала прогоняем лог через `scripts/live_run_table_validator.py`.
2. Затем публикуем JSONL строками через `scripts/publish_live_run_rows_to_sheet.py`.
3. **Каждый live/scenario/prod-smoke прогон, по которому делаем вывод или отчёт пользователю, обязан быть записан в Google Sheet.** Локальный/VPS лог без строки в таблице — это только черновик диагностики, а не завершённый прогон.
4. Если прогон был перепроверкой старого лога после фикса валидатора, его тоже публикуем отдельной явной версией, например `prod-verify-echo-YYYYMMDD-HHMM`, чтобы было видно, чем новая интерпретация отличается от старых строк.
5. Для удобства используем one-shot обёртку `scripts/live_quality_cycle.py`.
6. Параллельно собираем promptfoo test set по версии через `eval/nmbot-answer-quality/scripts/build_cases.py`: он обновляет текущий `tests/live-run-cases.yaml` и сохраняет архив в `eval/nmbot-answer-quality/tests/versions/live-run-cases.vN.yaml`.

### Критично про MCP-слой

- В `mcp`-колонке всегда должны быть **только два слоя**: что отправили (`mcp_contract`) и что вернулось (`mcp_response`).
- Counts/names/signals живут в validator JSONL / promptmaster verdict, но не раздувают колонку `mcp`.
- Если раньше колонка `mcp` содержала summary, такие строки считаются устаревшими для полной MCP-layer validation и должны быть переписаны новым форматом.
- Старые строки до фикса сырого `mcp_response` нельзя использовать как окончательное доказательство, что ответ опирался именно на MCP-данные: их нужно перепубликовать новой версией, если по ним принимается решение.
- Если ответ использует факт, которого нет в сыром `mcp_response`, такой ответ считается невалидным: `score = 0`, даже если текст красивый. Красота без MCP-доказательства не засчитывается.

### Команды

```bash
# Проверить лог и подготовить JSONL
python3 scripts/live_run_table_validator.py logs/live_model_run_2026-07-04_rerun_115218_weak_v3.txt --version v3 --jsonl-out logs/live_model_run_2026-07-04_rerun_115218_weak_v3.rows.v3.jsonl

# Если сравниваем prompt-size/speed, добавляем health snapshot
python3 scripts/nmbot_health.py --json > /tmp/nmbot_health.json
python3 scripts/live_run_table_validator.py logs/live_model_run_2026-07-04_rerun_115218_weak_v3.txt --version prompt-shortening-baseline --health-json /tmp/nmbot_health.json --jsonl-out logs/live_model_run_2026-07-04_rerun_115218_weak_v3.rows.prompt-shortening-baseline.jsonl

# Показать, что уйдёт в лист
python3 scripts/publish_live_run_rows_to_sheet.py logs/live_model_run_2026-07-04_rerun_115218_weak_v3.rows.v3.jsonl

# Реально записать в Google Sheet
python3 scripts/publish_live_run_rows_to_sheet.py logs/live_model_run_2026-07-04_rerun_115218_weak_v3.rows.v3.jsonl --write

# Полный цикл одной командой
python3 scripts/live_quality_cycle.py logs/live_model_run_2026-07-04_rerun_115218_weak_v3.txt --version v3 --write
```

### Правило регрессии

- если слой уже зелёный, не перезапускаем его без новой причины;
- новые строки всегда получают явный `version`;
- финальный отчёт по прогону нельзя закрывать словами «проверено/готово», пока соответствующая версия не опубликована в Google Sheet или пока пользователь явно не разрешил оставить прогон только в логах;
- для принятия дальше смотрим не только `score`, но и текст `оценка` глазами.
