# Card Presentation Rule

Короткий стандарт того, как LLM должна показывать карточку объекта в любом сценарии.

## Цель

Карточка должна читаться как живая презентация, а не как выгрузка полей.

## Правила

1. **Сначала смысл, потом детали.**
   Начинай с того, почему этот вариант вообще стоит смотреть.

2. **Один объект = один главный акцент.**
   Не перечисляй всё подряд; выбери 1–2 самых сильных факта.

3. **Если объектов несколько — сравнивай их между собой.**
   У каждого варианта должен быть свой выгодный угол.

4. **Каждый объект — отдельный блок.**
    Если вариантов несколько, не склеивай их в один абзац: у каждого ЖК должен быть свой визуальный блок с пустой строкой между блоками. Так ответ легче читать и считать.
    Если search/card уже дал 3+ подходящих варианта, shortlist должен сохранить 3 явных карточки/блока и не схлопываться до одного объекта без причины.

5. **Если нужен машинный разбор, `response` должен быть JSON-объектом.**
   Для 1+ ЖК верни `response.items` — массив объектов, где каждый элемент это один ЖК. Так удобнее считать варианты и строить красивую разметку.

6. **Пиши человеческую пользу.**
    Не “школы есть”, а “удобно для семьи, потому что рядом школы и сад”.

7. **Не повторяй одну и ту же форму.**
    Нельзя делать одинаковые карточки для всех объектов.

8. **Только подтверждённые факты.**
    Всё, что сказано, должно быть в MCP-карточке.

9. **Один финальный вопрос.**
    В конце — мягкий следующий шаг, без лишнего шума.

10. **Без технических слов.**
    Не использовать `regex`, `mcp`, `json`, “карточка сформирована” и похожее.

11. **Проверяй, продаёт ли карточка.**
    Если вариант не выглядит живым и убедительным, карточка ещё не готова.

12. **Причина у каждого объекта должна объяснять, а не только называть.**
     Если `reason` слишком короткий или похож на ярлык вроде `с отделкой`, это уже слабая карточка.

13. **`reason` — это мини-презентация, а не подпись.**
    В машинном JSON поле всё ещё называется `reason`, но по смыслу это `presentation_reason`: 1–2 живых предложения по формуле **факт → отличие → польза для клиента**. Нельзя ограничиваться перечислением “сдан, с отделкой, парк рядом”. Нужно объяснить: чем вариант выделяется среди других и почему это удобно именно в текущем запросе.

## Признаки хорошей карточки

- короткое человеческое вступление;
- у каждого объекта свой отдельный блок;
- у каждого объекта свой акцент;
- есть понятная польза для клиента;
- список не выглядит сухим;
- у `reason` не просто ярлык, а короткое объяснение;
- `reason` звучит как мини-презентация: факт → отличие → польза;
- финальный вопрос один и отдельным абзацем.

## Признаки плохой карточки

- сухой список полей;
- одинаковые описания у всех объектов;
- сравнение отсутствует, хотя объектов несколько;
- есть факты не из карточки;
- ответ звучит как техотчёт.
- `reason` похож на короткую подпись, а не на объяснение пользы.
- `reason` просто перечисляет факты через запятую и не объясняет, почему этот ЖК сильнее или удобнее клиенту.
