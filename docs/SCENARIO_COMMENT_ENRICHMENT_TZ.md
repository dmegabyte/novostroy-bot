# NMBOT / Ирина — ТЗ на сценарные комментарии к ЖК

Дата: 2026-07-02

Статус: спецификация реализована в `scripts/chat_tester_bot.py` и задеплоена на VPS; документ нужен как описание слоя и правил.

## 1. Зачем это делаем

Сейчас бот уже умеет показывать список ЖК и добавлять короткую пользу через `sales_phrase`, но польза часто строится вокруг самых простых фактов: отделка, готовность, цена, площадь.

Новая цель — научить Ирину давать комментарий под сценарий клиента:

- для семьи: парк, школа, сад, поликлиника, двор, площадь;
- для инвестиций: стартовая цена, готовность, отделка, транспорт;
- для жизни: метро, готовность, отделка, сервисы рядом;
- для бюджета: честный компромисс и понятный нижний порог;
- для быстрого переезда: сданный дом, ключи, отделка;
- для метро/транспорта: пешая доступность, понятный маршрут.

Пример целевого комментария:

```text
Рядом Мещерский парк и Чоботовский лес — будет проще чаще гулять с детьми на свежем воздухе.
```

Главное правило: комментарий строится только из фактов MCP/search. Если в фактах нет парка, школы, метро, поликлиники или двора — бот не говорит об этом.

## 2. Архитектура

```text
user_text
  → stage orchestrator
  → scenario detector
  → MCP/search with scenario enrichment
  → normalize facts
  → semantic card per ЖК
  → sales_phrase model writes one comment
  → validator
  → stage presenter assembles final answer
```

Практически это работает так:

- после первого списка бот может обогатить top-3 ЖК точечным MCP/search;
- если клиент выбрал ЖК, бот использует enriched card или делает короткий точечный enrichment перед ответом;
- если enrichment не пришёл, бот отвечает безопасным fallback из facts, без выдумок.

Разделение ответственности:

| Слой | Что делает |
|---|---|
| Orchestrator | решает стадию: первый список, выбранный ЖК, оператор, уточнение |
| Scenario detector | решает сценарий клиента: family, investment, budget, etc. |
| Search enrichment | просит MCP вернуть нужные поля под сценарий |
| Normalizer | приводит поля к клиентскому виду |
| Semantic card builder | готовит компактную карточку для модели |
| Sales phrase model | пишет один короткий комментарий |
| Validator | запрещает выдумки и плохой стиль |
| Stage presenter | собирает финальный ответ с отступами и вопросом |

## 6. Semantic card

Модель получает не сырой MCP JSON, а компактную карточку.

```json
{
  "stage": "first_list",
  "scenario": "family",
  "object_name": "ЖК «Лучи»",
  "facts": [
    "район: Солнцево",
    "рядом Мещерский парк и Чоботовский лес",
    "запланированы 4 детских сада и 2 школы",
    "есть квартиры с отделкой",
    "площади от 22,5 до 86,5 м²"
  ],
  "allowed_comments": [
    "рядом есть место для прогулок с детьми",
    "школы и сады рядом упрощают семейный быт",
    "отделка уменьшает ремонтные хлопоты"
  ],
  "forbidden": [
    "доходность",
    "ликвидность",
    "рост цены",
    "безопасный район",
    "лучший вариант",
    "выгодная покупка"
  ]
}
```

## 7. Prompt для sales_phrase

```text
Ты пишешь один короткий комментарий к ЖК для клиента.

Используй только facts и allowed_comments из INPUT.
Не добавляй новых фактов.
Не обещай доходность, рост цены, ликвидность, скидки, ипотеку.
Не используй рекламные слова: лучший, идеальный, выгодный, перспективный, премиальный.
Пиши живо, как консультант по новостройкам.
Одно предложение, до 160 символов.
Без приветствия, без вопроса, без markdown.

INPUT:
{{semantic_card}}

Верни JSON:
{
  "comment": "..."
}
```

## 8. Валидатор

Комментарий отклоняется, если:

- содержит факт, которого не было во входе;
- содержит запрещённые слова;
- содержит вопрос;
- содержит markdown;
- длиннее 160 символов;
- повторяет комментарий другого ЖК в том же ответе;
- говорит про парк/школу/метро/поликлинику, если этого нет в `facts`;
- использует слова `лучший`, `выгодный`, `идеальный`, `премиальный`, `перспективный`.

Fallback: если модель дала плохой комментарий, берём безопасную фразу из карты `fact → meaning`.

## 9. Интеграция в текущие стадии

### 9.1 first_list

```text
1. MCP нашёл 2–3 ЖК.
2. Для каждого ЖК собираем semantic card.
3. sales_phrase возвращает comment.
4. stage presenter собирает:
   - intro;
   - numbered list;
   - facts;
   - comment;
   - final question.
```

Финальный вопрос:

```text
Какой ЖК хотите рассмотреть подробнее?
```

### 9.2 selected_object

```text
1. Клиент выбрал ЖК.
2. Берём facts из last_options / last_search_response.
3. Если нужно, делаем enrichment по выбранному ЖК.
4. sales_phrase даёт один сильный комментарий.
5. presenter заканчивает операторским вопросом.
```

Финальный вопрос:

```text
Хотите, позвать оператора проверить актуальные квартиры по этому ЖК?
```

## 12. Acceptance criteria

Готово, если:

- каждый ЖК в списке имеет отдельный комментарий;
- комментарии не повторяются;
- семейные ответы используют инфраструктуру, если MCP её дал;
- бот не говорит про парк/школу/поликлинику, если MCP этого не дал;
- selected-object всегда ведёт к оператору;
- нет технических слов `MCP`, `в базе`, `по данным`, `сдача/готовность`;
- нет рекламных обещаний;
- все ответы имеют нормальные отступы;
- финальный вопрос ровно один.

## 13. Риски

1. MCP не всегда отдаёт инфраструктуру в broad search.
   - Решение: enrichment-запрос по точечному ЖК или расширение `search_v1.txt`.

2. Модель может красиво выдумать лишнее.
   - Решение: validator + fallback.

3. Комментарии могут повторяться.
   - Решение: отслеживать used comment types в одном ответе.

4. Ответы могут стать длинными.
   - Решение: один комментарий на ЖК, до 160 символов.

1. Сначала family-сценарий: парки, школы, сады, поликлиника.
