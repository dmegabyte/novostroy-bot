# Irina / MINION — карта диалога v1

Дата: 2026-07-02
Статус: рабочее ТЗ для новой итерации presenter-слоя

## Главная идея

Диалог не должен быть одним большим LLM-ответом на все случаи.

У Ирины есть стадии. На каждой стадии — своя задача, свой формат ответа и свой следующий шаг.

Коротко:

```text
1. Клиент просит подобрать квартиру
   → показываем 2-3 ЖК
   → вопрос: какой ЖК рассмотреть подробнее?

2. Клиент выбрал ЖК
   → коротко презентуем выбранный ЖК
   → вопрос: позвать оператора проверить актуальные квартиры?

3. Клиент просит наличие / этажи / корпуса / ипотеку / бронь / показ
   → это уже live-проверка
   → ведём к оператору

4. Клиент уточняет бюджет / отделку / район
   → фильтруем прошлый список или делаем новый MCP-поиск
   → снова показываем варианты или честно говорим, что близко к запросу

5. Клиент согласился на оператора
   → просим телефон
```

## Почему нужна карта

Сейчас плохие ответы появляются из-за смешения стадий:

- первый список пытается быть и подбором, и продажей, и операторской воронкой;
- выбранный ЖК иногда отвечает как обычная презентация, но не ведёт к оператору;
- модель видит сырые поля и пишет технические фразы вроде `сдача/готовность`;
- reason-layer пытается чинить текст поверх уже неправильной архитектуры.

Новая логика: сначала определить стадию, потом собрать ответ по правилам этой стадии.

# Stage 0. Приветствие / старт

## Когда

Клиент только начал диалог или нажал `/start`.

## Цель

Коротко сказать, что Ирина помогает подобрать новостройку, и попросить свободный запрос.

## Формат ответа

```text
Привет. Помогу подобрать новостройку в Москве или области.

Напишите, что ищете: район, бюджет, комнатность или цель покупки.
```

## Следующий шаг

Ждём свободный запрос клиента.

## Нельзя

- задавать длинную анкету;
- просить сразу телефон;
- предлагать оператора до пользы.

# Stage 1. Первый подбор / first_list

## Когда

Клиент пишет свободный запрос:

- `нужна двушка для семьи`;
- `квартира для инвестиций`;
- `студия до 8 млн`;
- `поближе к метро`;
- `квартира в Котельниках с отделкой`.

## Что делает система

1. Извлекает параметры: район, бюджет, комнатность, отделка, цель.
2. Делает MCP-поиск.
3. Берёт максимум 3 ЖК из `facts[]` или аккуратно из `near[]`.
4. Код готовит смысловые карточки объектов:
   - название;
   - локация;
   - цена;
   - отделка;
   - готовность / срок человеческим языком;
   - площадь;
   - разрешённая польза.
5. Модель или presenter превращает карточки в живой список.

## Цель ответа

Дать клиенту выбор из 2-3 ЖК и привести к выбору одного ЖК.

## Формат ответа

```text
Подобрала три варианта для семьи.

1. Город-парк «Переделкино Ближнее» — Внуковское поселение, дом уже сдан, есть квартиры с отделкой, площади от 26,8 до 97 м², цены от 9,56 млн рублей.
   Готовый дом проще планировать для переезда, а отделка уменьшает ремонтные хлопоты.

2. Жилой район «Скандинавия» — Коммунарка, дом уже сдан, есть квартиры с отделкой, площади от 19 до 153,3 м², цены от 10,74 млн рублей.
   Широкий диапазон площадей помогает подобрать формат под семью.

3. ЖК «Лучи» — Солнцево, дом уже сдан, есть квартиры с отделкой, площади от 22,5 до 86,5 м², цены от 10,89 млн рублей.
   С отделкой меньше ремонта после покупки, а готовый дом проще планировать для переезда.

Какой ЖК хотите рассмотреть подробнее?
```

## Правила

- Максимум 3 ЖК.
- Только нумерованный список `1./2./3.`.
- Каждый пункт: факты + одна человеческая польза.
- Польза только из разрешённых фактов:
  - `сдан / готов` → проще планировать переезд;
  - `с отделкой` → меньше ремонта;
  - `площадь` → проще подобрать формат под семью;
  - `цена` → понятно, от какого бюджета смотреть;
  - `метро` → удобнее каждый день ездить.
- В конце ровно один вопрос: `Какой ЖК хотите рассмотреть подробнее?`

## Нельзя

- предлагать оператора в первом списке, если варианты уже найдены;
- писать `сдача/готовность`;
- писать `верхняя точка бюджета`;
- писать `лучший`, `идеальный`, `выгодный`, `перспективный`, `премиальный`;
- выдумывать школы, парки, двор, безопасность, доходность, аренду, ипотеку, скидки;
- писать технические слова: `MCP`, `в базе`, `по данным`.

## Следующий шаг

Клиент выбирает ЖК по номеру или названию.

Переход: `Stage 1 → Stage 2`.

# Stage 1B. Первый подбор: точных вариантов нет, но есть близкие

## Когда

`facts=[]`, но `near[]` не пустой.

## Цель

Не бросить клиента, а показать близкие варианты и честно объяснить отличие.

## Формат ответа

```text
Точно под этот запрос вариантов не вижу, но есть близкие.

1. ...
   Отличие: ...

2. ...
   Отличие: ...

Хотите посмотреть один из этих вариантов или расширить условия?
```

## Правила

- Если запрос широкий — можно показать near как обычные варианты без слова “нет”.
- Если запрос узкий — честно сказать, чем near отличается.
- Не делать вид, что near — это точное совпадение.

## Следующий шаг

- Клиент выбирает близкий ЖК → `Stage 2`.
- Клиент уточняет условия → `Stage 4`.

---

# Stage 1C. Пустой результат

## Когда

`facts=[]` и `near=[]`.

## Цель

Не писать тупик, а предложить следующий полезный шаг.

## Формат ответа

```text
По этим условиям сейчас не вижу подходящих новостроек.

Могу расширить район, бюджет или посмотреть близкие варианты. Что поменяем?
```

## Если запрос вне региона

```text
Я помогаю с новостройками в Москве и Московской области.

Могу посмотреть варианты в Москве или рядом с областью. Где искать?
```

## Следующий шаг

Клиент меняет условия → `Stage 4`.

# Stage 2. Клиент выбрал ЖК / selected_object

## Когда

Клиент пишет:

- `ЖК Лучи`;
- `Скандинавия`;
- `второй`;
- `расскажи про Лучи`;
- `интересен первый`.

## Что делает система

1. Не делает новый широкий поиск.
2. Берёт выбранный объект из `state.last_options`.
3. Если есть `state.enriched_options`, берёт обогащённую карточку выбранного ЖК.
4. Если enrichment ещё не готов, делает короткий точечный MCP/search по выбранному ЖК.
5. Собирает презентацию из enriched card или из памяти.
6. Ведёт к оператору для проверки актуальных квартир.

## Цель ответа

Клиент уже заинтересовался объектом. Значит задача — не бесконечно рассказывать, а довести до проверки актуальных квартир.

## Формат ответа

```text
ЖК «Лучи» — проект ЛСР в Солнцеве, метро в 5 минутах пешком. Площади от 22,5 до 86,5 м², цены от 10,96 до 31,58 млн рублей.

Для семьи это удобно: рядом Мещерский парк и Чоботовский лес, а в проекте есть детские сады, школы и двор без машин.

Хотите, позвать оператора проверить актуальные квартиры по этому ЖК?
```

## Правила

- 2-3 коротких абзаца.
- Никакого нового списка ЖК.
- Никакого нового MCP-поиска, если выбранный ЖК уже есть в памяти.
- Если есть enriched card, использовать её.
- Если enrichment не готов, брать безопасную базовую карточку.
- Только факты выбранного ЖК.
- Финальный вопрос всегда ведёт к оператору.

## Нельзя

- заканчивать точкой без вопроса;
- писать `Показать, какие варианты сейчас есть в наличии?` — бот сам не показывает live-наличие;
- обещать наличие квартир;
- говорить `оператор свяжется`, пока клиент не согласился;
- снова продавать 3 ЖК.

## Следующий шаг

- Клиент говорит `да`, `позови`, `оператора`, `проверь` → `Stage 3`.
- Клиент спрашивает про другой ЖК → `Stage 2` по другому объекту.
- Клиент просит сравнить → `Stage 5`.

---

# Stage 3. Оператор / проверка актуальных квартир

## Когда

Клиент:

- выбрал ЖК и согласился на оператора;
- спрашивает наличие;
- спрашивает этажи, корпуса, планировки;
- спрашивает бронь, ипотеку, скидки, показ;
- просит “связаться”, “позвонить”, “менеджера”.

## Цель ответа

Получить согласие и телефон, чтобы оператор проверил актуальные квартиры.

## Формат ответа

```text
Да, это уже лучше проверить с оператором: он посмотрит актуальные квартиры, цены и условия по этому ЖК.

Оставьте номер телефона — передам, что интересует ЖК «Лучи».
```

## Если клиент уже дал согласие, но не дал телефон

```text
Напишите номер телефона, и оператор проверит актуальные квартиры по ЖК «Лучи».
```

## Если клиент отказался от оператора

```text
Хорошо, продолжим здесь.

Хотите сравнить этот ЖК с другими вариантами или изменить условия подбора?
```

## Правила

- Не просить телефон до согласия.
- Не обещать, что оператор уже всё сделал.
- Сохранять контекст: выбранный ЖК, цель, бюджет, комнатность.

## Следующий шаг

- Клиент даёт телефон → CRM/operator handoff.
- Клиент отказывается → `Stage 5` или `Stage 4`.

---

# Stage 4. Уточнение условий

## Когда

Клиент после списка пишет:

- `до 12 млн`;
- `только с отделкой`;
- `подешевле`;
- `готовый дом`;
- `поближе к метро`;
- `другой район`.

## Цель ответа

Не начинать диалог заново, а уточнить текущий подбор.

## Что делает система

1. Сначала пробует использовать `state.last_options`.
2. Если данных хватает — фильтрует/сортирует память.
3. Если данных не хватает — делает новый MCP-поиск с обновлёнными параметрами.

## Формат ответа, если хватило памяти

```text
По последнему списку подешевле выглядит ЖК «Переделкино Ближнее» — цены начинаются от 9,56 млн рублей.

Хотите рассмотреть его подробнее?
```

## Формат ответа, если нужен новый поиск

```text
Поняла, смотрю уже с новым условием: до 12 млн и с отделкой.

1. ...
2. ...
3. ...

Какой ЖК хотите рассмотреть подробнее?
```

## Следующий шаг

- Клиент выбирает ЖК → `Stage 2`.
- Клиент уточняет дальше → остаёмся в `Stage 4`.

---

# Stage 4.5. Ещё похожие варианты

## Когда

Клиент после списка пишет:

- `ещё варианты`;
- `похожие варианты`;
- `другие варианты`;
- `альтернативы`.

## Цель ответа

Расширить выбор, а не повторять уже показанный shortlist.

## Что делает система

1. Делает свежий MCP/search по тем же или близким условиям.
2. Исключает уже показанные ЖК из `visible_options` / `last_options`.
3. Показывает новый shortlist максимум из 3 вариантов.
4. Если новый список не получается — предлагает уточнить район, бюджет или формат.

## Формат ответа

```text
Хорошо, посмотрю ещё похожие варианты, но без повторов из прошлого списка.

1. ...
2. ...
3. ...

Какой из них разобрать дальше?
```

## Правила

- Не повторять тот же список один в один.
- Не превращать этот запрос в сравнение текущего списка.
- Если клиент пишет `сравни`, `чем отличаются`, `разница` — это `Stage 5`, а не этот шаг.

## Следующий шаг

- Клиент выбирает ЖК → `Stage 2`.
- Клиент просит сравнение → `Stage 5`.
- Клиент уточняет условия → `Stage 4`.

---

# Stage 5. Сравнение ЖК

## Когда

Клиент пишет:

- `сравни первый и второй`;
- `что лучше из этих`;
- `какой дешевле`;
- `какой быстрее сдан`.
- `чем отличаются`.

## Цель ответа

Сравнить только по реальным полям, не выбирать “лучший” от себя.

## Формат ответа

```text
Если сравнивать по фактам:

1. ЖК «Переделкино Ближнее» — ниже старт по цене и есть отделка.

2. Жилой район «Скандинавия» — шире диапазон площадей, тоже есть отделка.

Для семьи я бы сначала разобрала тот, где удобнее площадь и бюджет.

Какой из двух хотите рассмотреть подробнее?
```

## Правила

- Не говорить “лучший”.
- Не выдумывать преимущества.
- Сравнивать только по цене, площади, отделке, готовности, метро, локации — если они есть.

## Следующий шаг

Клиент выбирает ЖК → `Stage 2`.

---

# Stage 6. Клиент дал телефон

## Когда

Клиент прислал номер после согласия на оператора.

## Цель ответа

Подтвердить, что оператор получит контекст.

## Формат ответа

```text
Приняла номер.

Передам оператору, что вас интересует ЖК «Лучи» и нужно проверить актуальные квартиры по нему.
```

## Правила

- Не раскрывать внутренности CRM.
- Не обещать точное время звонка, если его нет.
- Передать контекст: ЖК, цель, бюджет, комнатность.

# Общая карта переходов

```text
START
  ↓
Stage 0: приветствие
  ↓
Stage 1: первый подбор
  ├─ есть 2-3 ЖК → вопрос “Какой ЖК?” → Stage 2
  ├─ есть близкие → выбрать / уточнить → Stage 2 или Stage 4
  └─ ничего нет → расширить условия → Stage 4

Stage 2: выбран ЖК
  ├─ согласие на оператора / live-детали → Stage 3
  ├─ другой ЖК → Stage 2
  ├─ сравнить → Stage 5
  └─ изменить условия → Stage 4

Stage 3: оператор
  ├─ дал телефон → Stage 6
  └─ отказался → Stage 5 или Stage 4

Stage 4: уточнение условий
  ├─ нашли варианты → Stage 1
  └─ выбрал ЖК → Stage 2

Stage 5: сравнение
  └─ выбрал ЖК → Stage 2

Stage 6: телефон принят
  └─ конец клиентского сценария / операторская обработка
```

# Stage orchestrator: как решать текущую стадию

Оркестратор должен срабатывать **до генерации текста**.

Он не пишет ответ. Его задача — вернуть одно решение:

```json
{
  "stage": "first_list | selected_object | expand_more_options | operator_handoff | refinement | comparison | empty_result | phone_capture | greeting",
  "reason": "почему выбрана стадия",
  "selected_option": "если есть",
  "needs_mcp_search": true,
  "presenter": "какой renderer использовать"
}
```

Главное правило: **семантику распознаёт оркестратор**.

Он нужен именно для выбора сценария по смыслу, а не по одному слову. Нижние regex/router не должны решать, что клиент “на самом деле хотел”.

Кодом определяется только механика без семантики:

- номер телефона.

Всё остальное — LLM-orchestrator: выбор варианта по `1`/`2`/`3` или названию ЖК, похожие варианты, отказ, согласие на оператора, смена условий, уточнение бюджета, просьба “другой/ещё/подешевле”.
Если клиент выбирает номером, orchestrator смотрит на `visible_options` и возвращает exact `selected_option_name`; код не выбирает ЖК regex’ом.

## Входы оркестратора

```text
user_text              — последнее сообщение клиента
state.last_options     — прошлый список ЖК
state.visible_options  — ЖК, реально показанные клиенту в последнем списке
state.selected_option  — выбранный ЖК, если уже есть
state.last_offer_type  — что бот предложил последним сообщением
state.awaiting_phone   — ждём ли телефон
state.params           — цель, бюджет, район, комнатность
state.turns_after_results
```

## Приоритеты решения

Порядок важен. Чем выше правило, тем раньше оно проверяется.

### 1. Ждём телефон

Если `state.awaiting_phone = true`:

- если сообщение похоже на телефон → `phone_capture`;
- если клиент отказывается → `operator_reject`;
- иначе → `phone_clarification`.

```json
{
  "stage": "phone_capture",
  "presenter": "render_phone_accepted",
  "needs_mcp_search": false
}
```

### 2. Клиент согласился на оператора после выбранного ЖК

Если есть `state.selected_option`, а клиент пишет `да`, `позови`, `оператор`, `менеджер`, `проверь`, `хочу`, `давайте` после вопроса про оператора:

```json
{
  "stage": "operator_handoff",
  "selected_option": "state.selected_option",
  "presenter": "render_operator_contact_request",
  "needs_mcp_search": false
}
```

### 3. Клиент просит live-детали по выбранному ЖК

Если есть `state.selected_option` и в тексте есть:

```text
наличие, актуальные, квартиры, этаж, корпус, планировка,
бронь, забронировать, ипотека, ставка, скидка, показ,
посмотреть, менеджер, оператор, звонок
```

→ это не презентация, а операторская стадия.

```json
{
  "stage": "operator_handoff",
  "selected_option": "state.selected_option",
  "presenter": "render_operator_handoff",
  "needs_mcp_search": false
}
```

### 4. Клиент выбрал ЖК из списка

Если по смыслу сообщения клиент выбрал номер варианта (`1`, `2`, `второй`) или название ЖК из `visible_options/last_options`, это распознаёт LLM-orchestrator:

```json
{
  "stage": "selected_object",
  "selected_option": "exact name from visible_options",
  "presenter": "render_selected_object",
  "needs_mcp_search": false
}
```

Важно: выбор ЖК — это **не новый поиск**. Ответ берётся из памяти.

### 5. Клиент просит сравнить

Если есть прошлый список и текст содержит:

```text
сравни, отличие, разница, что лучше, какой дешевле,
какой быстрее, первый или второй, чем они отличаются
```

```json
{
  "stage": "comparison",
  "presenter": "render_comparison",
  "needs_mcp_search": false
}
```

### 5.5. Клиент просит ещё похожие варианты

Если есть прошлый список и клиент по смыслу просит расширить подбор, например:

```text
подбери похожие, найди похожие, покажи похожие,
ещё такие, ещё варианты, другие варианты,
альтернативы, можно похожие, давай ещё
```

Это **не comparison** и **не continue_selection**.

```json
{
  "stage": "expand_more_options",
  "presenter": "render_first_list",
  "needs_mcp_search": true,
  "visible_options_policy": "rebuild"
}
```

Правила:

- использовать прошлые `state.params` и смысл последнего подбора;
- исключить уже показанные `state.visible_options` / `state.last_options`;
- не повторять те же три ЖК;
- не задавать уточняющий вопрос, если смысл “хочу ещё похожие” ясен;
- не переводить это в `continue_selection`, потому что `continue_selection` работает по памяти и может закольцевать клиента.

Плохой сценарий:

```text
Клиент: подбери похожие
Бот: Уточните, продолжить подбор или изменить условия?
Клиент: подбери похожие
Бот: Хорошо, продолжим подбор... <те же три ЖК>
```

Почему плохо: оркестратор не выбрал semantic stage. Это ошибка stage selection, а не проблема текста presenter’а.

### 6. Клиент уточняет условия прошлого подбора

Если есть `last_options`, а клиент пишет:

```text
до 12 млн, подешевле, дороже, с отделкой, без отделки,
готовый, сдан, другой район, поближе к метро, студия, двушка
```

Сначала пробуем ответить по памяти.

```json
{
  "stage": "refinement",
  "presenter": "render_refinement",
  "needs_mcp_search": false
}
```

Если в памяти нет нужных полей или уточнение меняет область поиска — тогда:

```json
{
  "stage": "first_list",
  "presenter": "render_first_list",
  "needs_mcp_search": true
}
```

### 7. Новый свободный запрос

Если нет выбранного ЖК, нет ожидаемого телефона, и сообщение похоже на новый поиск:

```text
квартира, студия, однушка, двушка, трёшка,
для семьи, для инвестиций, район, метро, бюджет
```

```json
{
  "stage": "first_list",
  "presenter": "render_first_list",
  "needs_mcp_search": true
}
```

### 8. Непонятное сообщение

Если стадия не распознана:

```json
{
  "stage": "clarification",
  "presenter": "render_clarification",
  "needs_mcp_search": false
}
```

Пример ответа:

```text
Подскажите, вы хотите выбрать один из прошлых ЖК или изменить условия подбора?
```

| Ситуация | Условие | Stage | Presenter | MCP |
|---|---|---|---|---|
| Первый запрос | нет памяти, запрос про квартиру | `first_list` | `render_first_list` | да |
| Есть 2-3 ЖК, клиент пишет `2` | номер из списка | `selected_object` | `render_selected_object` | нет |
| Есть 2-3 ЖК, клиент пишет название | match по `visible_options` | `selected_object` | `render_selected_object` | нет |
| Клиент выбрал ЖК и пишет `наличие?` | live-data trigger | `operator_handoff` | `render_operator_handoff` | нет |
| Клиент выбрал ЖК и пишет `да` после вопроса про оператора | consent | `operator_handoff` | `render_operator_contact_request` | нет |
| Бот ждёт телефон, клиент прислал номер | phone-like text | `phone_capture` | `render_phone_accepted` | нет |
| После списка клиент пишет `подешевле` | refinement | `refinement` | `render_refinement` | сначала нет |

| После списка клиент пишет `сравни 1 и 2` | compare | `comparison` | `render_comparison` | нет |
| Нет вариантов | `facts=[]`, `near=[]` после MCP | `empty_result` | `render_empty_result` | уже был |

## Главное правило для оркестратора

Оркестратор решает **не “что ответить”, а “какой presenter имеет право отвечать”**.

```text
stage → presenter → validator → response
```

Если выбран неправильный stage, даже хороший prompt даст плохой ответ.

Пример:

- `ЖК Лучи` после списка — это `selected_object`, а не `first_list`.
- `Есть квартиры в наличии?` после выбранного ЖК — это `operator_handoff`, а не `selected_object`.
- `до 12 млн` после списка — это `refinement`, а не новый старт диалога.
- `подбери похожие` после списка — это `expand_more_options`, а не `clarification`, не `continue_selection` и не regex-ветка.

Если для исправления semantic stage хочется добавить regex, значит выбран не тот слой. Надо уточнять контракт orchestrator’а и проверять его exact live-фразой.

# Что должно быть в коде

Минимально нужны не “ещё один промпт”, а stage-presenter функции:

```text
detect_stage(message, state) -> stage
normalize_mcp_options(search_response) -> options
build_semantic_cards(stage, options, state) -> cards
render_first_list(cards) -> response
render_selected_object(card) -> response
render_operator_handoff(selected_object, state) -> response
render_refinement(cards/state) -> response
render_comparison(cards) -> response
validate_response(stage, response) -> ok/error
```

LLM можно оставить только внутри `render_first_list` и `render_selected_object`, но она должна получать уже готовые смысловые карточки, а не сырые поля.

# Главные acceptance criteria

1. Первый список всегда заканчивается вопросом: `Какой ЖК хотите рассмотреть подробнее?`
2. После выбора ЖК ответ всегда ведёт к оператору: `Хотите, позвать оператора проверить актуальные квартиры по этому ЖК?`
3. В клиентском тексте нет `сдача/готовность`, `верхняя точка бюджета`, `MCP`, `в базе`, `по данным`.
4. В первом списке каждый ЖК получает отдельную пользу.
5. Оператор не предлагается в первом списке, если варианты уже найдены.
6. Оператор предлагается после выбора ЖК или вопроса про live-детали.
7. Бот не обещает наличие квартир, этажи, корпуса, скидки, ипотеку — только оператор может проверить это.
8. Если данных нет, бот не выдумывает, а предлагает расширить условия или подключить оператора.
