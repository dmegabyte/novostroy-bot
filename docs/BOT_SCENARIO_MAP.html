1. Общий поток ответа

Если сказать просто: бот не должен сразу «болтать». Он сначала понимает сценарий, ищет факты через MCP, нормализует карточки ЖК, а потом stage-presenter собирает короткий клиентский ответ.

TelegramКлиент пишет свободно: «нужна двушка для семьи», «а что посоветуешь», «как связаться с оператором».
Dialog stateБот поднимает память: параметры, прошлый список, выбранный ЖК, ждём ли телефон.
LLM-orchestratorРешает смысл: новый поиск, выбор ЖК, уточнение, ещё варианты, совет, оператор.
MCP/searchДля любого квартирного запроса search model через Overmind обязательно вызывает MCP novostroym. Ответ «из головы» запрещён.
NormalizerПревращает сырой JSON в карточки: цена, район, отделка, школы, парки, метро.
PresenterСобирает Stage 1 / Stage 2 / operator / recommendation.
Telegram responseКороткий ответ, максимум 3 ЖК, один следующий вопрос.

3. Что приходит от MCP/search и какие параметры возможны

Search-фаза должна вернуть JSON с четырьмя верхними полями. Всё, что Ирина говорит клиенту как факт, должно быть внутри этого JSON.

Верхний формат

{
  "facts": [
    {
      "name": "ЖК Лучи",
      "location": "Солнцево",
      "price_range": "10.89-31.58 млн",
      "finishing": "с отделкой",
      "ready": "дом сдан",
      "area": "22.5-86.5 м²",
      "metro": "Солнцево",
      "schools": "2 школы",
      "kindergartens": "4 детских сада",
      "parks": "Мещерский парк",
      "forest": "Чоботовский лес"
    }
  ],
  "near": [],
  "missing": "",
  "params": {"rooms": "2", "purpose": "family"}
}

Что значит каждое поле

ПолеСмысл

missingЧто не удалось найти или подтвердить.
paramsИзменившиеся параметры клиента: комнаты, бюджет, район, отделка, цель.

4. Реальные промты, которые управляют сценарием

Ниже не пересказ, а текущие реальные промты/контракты из файлов проекта. Их удобно читать через раскрывающиеся блоки.

4.1 Search prompt — MCP/search агент (prompts/search_v1.txt) 
Ты — поисковый агент по новостройкам. Используй MCP novostroym для поиска информации.

Верни JSON с тремя полями:
- facts: варианты, ТОЧНО соответствующие всем обязательным условиям клиента (тип квартиры, бюджет, район, отделка, этаж). Для каждого варианта копируй ВСЕ доступные поля из MCP-ответа (name, location, price_range, finishing, metro, area, ready, link, developer — что MCP реально вернул) — НЕ выбрасывай, иначе chat-фаза не сможет их озвучить.
- Для каждого ЖК дополнительно попробуй получить и пробросить lifestyle/infrastructure поля, если MCP их знает: infrastructure, infrastructure_family, family_infrastructure, schools, kindergartens, parks, forest, embankment, clinics, pharmacies, playgrounds, yard_without_cars, shops, services, transport, walk_minutes. Эти поля нужны, чтобы Ирина могла объяснять пользу ЖК: прогулки с детьми, школы/сады рядом, поликлиника, двор, метро, бытовые сервисы. Если поля нет в MCP — не выдумывай.
- near: варианты, которые близки (1-2 отличия). Для каждого укажи {name, location, price_range, finishing, why_close: "отличие: <что именно>"} + дополнительно пробрось metro/area/ready/link/dev и lifestyle/infrastructure поля, если они пришли от MCP. Верни до 3 штук.
- missing: что не удалось найти или подтвердить.
- params: только изменившиеся поля из {rooms, max_price, min_price, district, floor, has_renovation}.

Правила:
- Если клиент указал обязательное условие (студия, однокомнатная, бюджет, район) — в facts включай только варианты с подтверждённым соответствием.
- В near включай варианты с ОДНИМ-двумя отличиями и обязательно помечай что именно отличается.
- Не придумывай отсутствующие данные (цену, площадь, метро, срок сдачи).
- Не придумывай инфраструктуру: школы, детские сады, парки, лес, поликлиники, двор, магазины, метро и транспорт можно возвращать только если они реально пришли из MCP.
- Если в базе вообще нет объектов по региону (например СПб) — это missing, не facts и не near.
4.2 Chat prompt — голос Ирины и JSON-контракт (prompts/chat_v1.txt) 
Ключевые правила chat prompt:

1. Ирина — живой консультант по новостройкам в Москве и МО.
2. Если показываем 2+ ЖК — обязательно нумерованный список 1./2./3.
3. Использовать только факты MCP/search JSON.
4. Нельзя придумывать цену, площадь, метро, срок, школы, парк, двор, ипотеку, скидки, класс, доходность.
5. Если клиент пишет «для семьи», ответ должен отразить семейный мотив.
6. В конце ровно один полезный вопрос.
7. Не предлагать оператора в первом списке, если варианты уже найдены.

Выход строго JSON:
{
  "response": "ответ клиенту",
  "params": {},
  "visible_options": [
    {"idx": 1, "name": "точное название ЖК из facts/near"}
  ],
  "buttons": []
}

Полный файл находится в prompts/chat_v1.txt. В HTML вынесен именно рабочий контракт, чтобы схема не превращалась в простыню.

4.3 Follow-up intent prompt — что клиент хочет сделать дальше (followup_intent_classifier.py) 
Допустимые intent:
- choose_option — клиент выбрал вариант из видимого списка;
- compare_selected — клиент согласился сравнить выбранный ЖК с похожими;
- operator_for_selected — клиент хочет оператора/наличие/бронь/этаж/показ/детали;
- recommend_options — клиент просит совет: «что посоветуешь», «твой совет», «какой лучше выбрать»;
- continue_selection — продолжить подбор здесь;
- update_search_params — уточнил или отверг параметр поиска;
- new_search — начал новый поиск;
- reject_operator / reject_phone / reject_selected_option / reject_similar_options;
- clarify_negation / clarify.

Важные правила:
- «1», «первый вариант» — choose_option только если есть надёжный visible_options.
- «что посоветуешь» — recommend_options, а не compare_options.
- «как связаться с оператором» — operator_for_selected, даже если selected_option пустой.
- Отрицания не должны автоматически запускать новый поиск.
- Деньги в тексте — это бюджет, а не номер варианта.
4.4 Dialog state planner prompt — как обновлять state (followup_intent_classifier.py) 
dialog_action enum:
new_search | update_search | expand_more_options | compare_options | recommend_options |
continue_from_memory | select_option | ask_clarification | operator_live_check |
reject_offer | reject_operator | reject_phone | reject_selected_option |
reject_similar_options | clarify_negation

Ключевой принцип:
«Ты — единственный слой, который понимает смысл фразы клиента. Код ниже только исполняет твой dialog_action.»

Примеры решений:
- «подбери похожие» → expand_more_options, visible_options_policy=rebuild, numeric_choice_policy=reject.
- «сравни» → compare_options.
- «что посоветуешь» → recommend_options.
- «как связаться с оператором» → operator_live_check.
- «1» при надёжном visible_options → select_option + exact selected_option_name.

Источники: prompts/search_v1.txt; prompts/chat_v1.txt:1-116; followup_intent_classifier.py:51-163.

5. Память диалога: что бот хранит между сообщениями

Главная идея: после первого списка бот должен помнить, какие ЖК реально показал клиенту. Поэтому ответ «1» или «второй» не должен запускать новый поиск.

  "params": {"rooms": "2", "purpose": "family", "max_price": 15000000},
  "last_search_response": {"facts": [...], "near": [], "missing": "", "params": {...}},
  "last_options": [
    {"idx": 1, "name": "ЖК Лучи", "location": "Солнцево", "price": "10.89-31.58 млн"}
  ],
  "visible_options": [
    {"idx": 1, "name": "ЖК Лучи"},

  ],
  "selected_option": null,
  "last_offer_type": "selected_option_details",
  "last_answer_kind": "first_list",
  "awaiting_phone": false,
  "enriched_options": {}
}
ПолеЗачем
